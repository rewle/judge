# AGENTS.md

## Что это

`judge` — CI/CD-гейты для реестра Claude Code скиллов (и похожих конфигов):
каждый PR со скиллом должен пройти цепочку проверок, прежде чем попасть в
реестр. Не привязан к конкретному реестру — предназначен для внешних
целевых систем, не для этого репозитория и не для личного Obsidian vault
владельца.

Мотивация: ручной ревью каждого PR со скиллом не масштабируется (дубли,
избыточные права, галлюцинации, раздутые токены, уязвимость к prompt
injection).

## Архитектура

Гейты прогоняются **последовательно**, порядок и состав — `config.yaml`
(`gates:`), это единственный источник истины по порядку, а не README/эта
инструкция. Fail любого гейта останавливает цепочку.

Два типа гейтов:
- **per-skill** (01, 02, 03, 04, 05, 06, 08) — сигнатура `check(skill_path)`.
- **реестровый** (07, граф `uses:`) — сигнатура `check_registry(registry_dir)`,
  архитектурно отдельная, запускается через `--check-chains` или
  автоматически после чистого прогона 01-06/08 (`run_gates.py` без
  `--skill`/`--path`, если хоть один скилл не прошёл — 07 пропускается с
  явным сообщением, не гоняется вслепую).

Детерминированные гейты (01, 02, 08, часть 03/07) не требуют judge и должны
гонять на каждый push. Judge-зависимые слои (04, 05, 06, 03 v1, 07
I/O-совместимость/токен-бюджет/injection) дороже — задуманы для
`ready-for-review`/батча, не на каждый коммит. У каждого гейта — контракт в
`docs/<NN>_<name>.md`: вход, логика, порог, выход. Смотреть туда, а не
угадывать поведение по коду.

Судья (`judge_client.py`) — два бэкенда через `JUDGE_BACKEND`:
- `api` (дефолт, нужен для CI) — настоящий Anthropic API, платный ключ
  только через env (`JUDGE_API_KEY`/`ANTHROPIC_API_KEY`), в репозиторий не
  коммитить.
- `cli` (только локальная обкатка) — шелл-аут в `claude -p ...
  --json-schema ... --no-session-persistence`, использует локальную сессию
  вместо платного ключа. Не для CI (невоспроизводимо на чужой машине).
  **Гейт 05 через cli не работает** — `claude -p` не даёт offline-подсчёт
  токенов без генерации, что и есть ядро логики гейта.

## Структура репозитория

```
gates/        — код гейтов, один модуль на гейт
docs/         — контракт каждого гейта + roadmap_chains.md
policies/     — allowlist прав (+ systems/*.yaml — политика на систему),
                критерии рубрики, библиотека redteam-атак
examples/     — скиллы-фикстуры (по одной на гейт/сценарий) + expected_results.yaml (оракул)
                + chains/ и groups/ — реестры-фикстуры гейта 07 (без machine-оракула)
                + policies/ — фикстуры битых политик (--check-policy <корень>,
                оракул expected_policy_results.yaml; в живой policies/ не класть)
history/      — append-only логи (дрифт рубрики, кэш дублей/токен-бюджета)
config.yaml   — порядок гейтов, пороги, пути
run_gates.py  — раннер (fail-fast, --skill/--path/--check-chains/--skip-chains)
new_skill.py  — генератор скелета скилла для новичков
judge_client.py — оба бэкенда судьи
behavioral.py — agent-run + coverage-judge (используется гейтами 05 и 06)
scenario_emulator/ — эмулятор пользователя по декларативному сценарию —
                НЕ гейт из config.yaml, отдельный инструмент, см. ниже
docker/       — образ/entrypoint сендбокс-relay эмулятора сценариев
run_scenario.py — CLI эмулятора сценариев (отдельно от run_gates.py)
```

## Эмулятор сценариев (не гейт)

Отдельная подсистема, архитектурно независимая от цепочки гейтов
01-07/`run_gates.py`: по декларативному текстовому сценарию
(`examples/scenarios/*.md`) эмулирует пользователя в диалоге с внешним
«стендом», записывает трейс, проверяет его точно (`exact_checks`, без LLM)
и семантически (judge). Практическое руководство — `scenario_emulator/README.md`;
полный контракт и история решений — `docs/09_scenario_emulator.md` +
`docs/TODO_scenario_emulator.md` (там же — почему не гейт: решение 8). CLI
— `run_scenario.py`, регрессия фикстур — skill `/scenario-regress` (не
`/judge-regress` — тот только про
`run_gates.py`). Не путать `scenario_emulator/` с `behavioral.py`
(agent-run для гейтов 05/06) — разные механизмы: `behavioral.py` гоняет
сам скилл через judge на один запрос, эмулятор ведёт многоходовый диалог
эмулированного пользователя с внешним стендом. Третий пример —
`open-deposit` (`examples/scenarios/open-deposit.md`) — референс паттерна
детерминированного подтверждения критического действия (widget-callback
мимо модели, харнесс сам зовёт тул и сверяет `confirmation_id` по своему
стейту, а не полагается на то, что модель верно поймёт текст); стенд —
три сервиса в docker compose, который `run_scenario.py` поднимает и
убирает сам по полю `compose_file` из frontmatter сценария.

## Ключевые дизайн-решения — не пересматривать без явного запроса

- **Не обобщать дизайн по личному Obsidian vault владельца
  (`~/Projects/mind`).** Этот vault не репрезентативен для целевых систем,
  под которые пишется `judge` — использовать его можно только как
  иллюстрацию/фикстуру, не как источник истины о том, что «реально
  происходит» в целевых системах. Если для решения нужны данные о целевой
  системе, а их нет — держать несколько гипотез живыми и спросить
  владельца, а не подставлять этот vault как прокси. (Прецедент: роадмап
  `docs/roadmap_chains.md` один раз уже переписывался по ошибке на этом
  основании и был откачен.)
- Конвенции `uses: [skill-a, ...]` и `provides:` (свободный текст) во
  frontmatter — придуманы и спроектированы в рамках `judge`, не заданы
  извне. Не менять форму (например, добавлять `expects:` или
  `when:`/`type:` к `uses:`) без обсуждения — сознательно оставлены плоскими
  (YAGNI, нечего калибровать). Исключение, согласованное владельцем
  2026-07-13 (см. `docs/design_multisystem_groups.md`, решения 1C/2B/3B):
  `justification:` — строка ИЛИ карта `{tool_or_pattern: reason}`; политика
  прав — глобальный файл + `policies/systems/<system>.yaml` (файл на
  систему, валидируется «гейтом для политик»); опциональный проверяемый
  манифест группы `group.yaml` в папке реестра (гейт 07). Дальнейшие
  изменения формы — по-прежнему только через обсуждение.
- Пороги в `config.yaml` **не откалиброваны** без живого платного
  API-ключа: `rubric_min_score`, `rubric_max_criterion_sigma`,
  `token_efficiency_max_overhead_pct`,
  `token_efficiency_min_coverage_gain_pp`, `duplication_semantic_similarity`,
  `rubric_drift_warn_abs`, `chain_io_compat_min_score`,
  `chain_token_budget_max`. Не «улучшать» их на глазок — калибровка это
  отдельный шаг: прогнать 03(v1)/04/05/06(adversarial)/07 на всех
  фикстурах через `JUDGE_BACKEND=api`, сверить с
  `examples/expected_results.yaml`.
- Не тащить внешние раннеры для redteam (garak/PyRIT/promptfoo) — решение
  уже принято в пользу своего harness (`gates/g06_redteam.py`).

## Как запускать

```bash
python3 -m venv .venv && source .venv/bin/activate  # уже создано, python3.9
pip install -r requirements.txt

python3 run_gates.py                       # 01-06+08 по examples/skills, 07 автоматически если все прошли
python3 run_gates.py --skill golden-skill  # один скилл, без гейта 07
python3 run_gates.py --path ./my-skill     # скилл вне skills_dir
python3 run_gates.py --skip-chains         # без автозапуска гейта 07
python3 run_gates.py --check-chains --registry examples/chains  # только гейт 07
python3 run_gates.py --skills-dir ./my-group  # полный цикл по произвольной папке-группе
python3 run_gates.py --check-policy           # только валидация политики прав + config.yaml
```

Без `JUDGE_API_KEY`/`ANTHROPIC_API_KEY` и без `JUDGE_BACKEND=cli` gates
04/05/06(adversarial)/07(I/O, токен-бюджет) возвращают `not_configured` —
это ожидаемо, не баг, не блокирует остальную цепочку.

Нет отдельного test-раннера (pytest и т.п.) — фикстуры в `examples/` и
`run_gates.py` против них это и есть тесты. Перед коммитом в `gates/`
прогонять `python3 run_gates.py` (полная регрессия) и убеждаться, что
поведение на существующих фикстурах не изменилось, если это не было целью
правки.

эскалация прав, обратная проверка, I/O-совместимость, токен-бюджет цепочки,

прогоном через `JUDGE_BACKEND=cli`, но не через платный `api`-бэкенд.
**Единственный настоящий блокер:** ключ Anthropic API для калибровки порогов
(см. выше) и финальной проверки через `api`-бэкенд. После этого — либо
калибровка, либо интеграция с реальным реестром скиллов/CI.

Отдельно (не гейт, не блокирует статус выше): эмулятор сценариев — Фазы 0-3
реализованы и проверены живыми прогонами (диалог, точные проверки,
семантика, оба сендбокс-пути), Фаза 4 (фикстуры/CLI/доки) в процессе, см.
«Эмулятор сценариев» выше и `docs/TODO_scenario_emulator.md`.

## Правила для агента в этом репозитории

- Новые файлы, документация, фикстуры — добавлять свободно.
- Изменения в `gates/*.py` и `config.yaml` (особенно пороги и порядок
  гейтов) — влияют на поведение всей цепочки, проверять регрессией перед
  коммитом.
- Деструктивные операции (удаление фикстур/истории `history/*.jsonl`,
  `git reset --hard`, force-push, удаление веток) — только с явного согласия
  владельца.
- Пуш в `github.com/rewle/judge` (публичный репозиторий) — подтверждать с
  владельцем перед пушем, форс-пуш не использовать.
