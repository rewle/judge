# Гейт 07 — chain (граф `uses:`)

**Детерминизм:** 4 проверки детерминированные, судейские слои (I/O,
токен-бюджет, injection) — нет.
**Реализация:** `gates/g07_chain.py`
**Когда гонять:** реестровый — `--check-chains` или автоматически после
чистого полного прогона 01-06/08 (`--skip-chains` отключает).

Этот файл — контракт гейта (вход/логика/порог/выход), как у остальных
гейтов в `docs/<NN>_<name>.md`. Дизайн-история, обоснования решений и
разбор альтернатив — в первоисточнике `docs/roadmap_chains.md` (гейт
вырос из него и долго был задокументирован только там).

## Вход
Весь реестр (`registry_dir`), а не один скилл: сигнатура
`check_registry(registry_dir) -> {skill_name: GateResult}` —
архитектурно отдельная от per-skill `check(skill_path)`.

## Логика
Детерминированные проверки (без judge, на каждый push):
1. **Циклы** в графе `uses:`.
2. **Битые ссылки** — `uses:` на несуществующий в реестре скилл.
3. **Эскалация прав через композицию** — объединение `tools:` по
   транзитивному замыканию `uses:` прогоняется через тот же движок
   политики, что гейт 02 (`evaluate_tools`): ловится «A с виду просит
   только Read, но через `uses: [B]` эффективно получает Bash(*)».
   Сообщение называет каждый унаследованный инструмент и его
   происхождение.
4. **Необъявленные вызовы** — тело упоминает скилл реестра
   (backtick-спан), которого нет в `uses:`.

Судейские слои (нужен judge; без ключа — пропускаются, видны только
4 проверки выше):
5. **I/O-совместимость** — judge оценивает, стыкуется ли `provides:`
   поставщика с ожиданиями потребителя.
6. **Токен-бюджет цепочки** — сумма input-токенов скилла + транзитивных
   `uses:` (только api-бэкенд: cli не умеет count_tokens; кэш
   `history/token_budget_cache.json`, ключ `model:hash(text)`).
7. **Многошаговый injection** — payload, внедрённый через выход
   поставщика, против потребителя.

## Манифест группы (решение 2B, проверяемый)
Если в папке реестра лежит `group.yaml` — гейт сверяет декларацию с
фактом; манифест не может врать. Отсутствие файла — прежнее поведение
байт-в-байт.

```yaml
name: dwh-reporting
members: [dwh-loader, dwh-reporter]   # ровно содержимое папки
systems:
  dwh: read          # режим — контекст для ревьюера, не верифицируется
reviewer_note: |
  Группа только читает витрины DWH.
```

Проверяется: `members` ↔ фактические скиллы папки (оба направления —
FAIL); ключи `systems` ↔ фактические MCP-системы из `tools:` всех членов
(оба направления — FAIL); форма (неизвестные ключи, типы). Значение
режима (`read`/`send`/...) не верифицируется — это заметка ревьюеру, но
если инструменты группы подпадают под `always_flag_for_review`, строка
`group.yaml` заранее говорит «группа гарантированно потребует ручного
ревью» с перечнем. Результат — отдельной строкой `group.yaml` в отчёте
гейта. Фикстуры: `examples/groups/group-ok`, `examples/groups/group-desync`
(оракула нет, как у `examples/chains`, — ожидания в комментариях
`group.yaml` фикстур).

## Пороги
`chain_io_compat_min_score` (не откалиброван), `chain_token_budget_max`
(не откалиброван) — см. `config.yaml` и правило калибровки в `AGENTS.md`.

## Выход
`{skill_name: GateResult}` на каждый скилл реестра с валидным
frontmatter; FAIL любого скилла — ненулевой код выхода раннера. Если
сам файл политики сломан (см. `docs/02_permissions.md`, «Валидность
самой политики»), гейт возвращает одну FAIL-строку, атрибутированную
`policies/tools_allowlist.yaml`, а не по строке на скилл.
