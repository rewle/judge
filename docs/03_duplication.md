# Гейт 03 — duplication

**Детерминизм:** v0 полностью детерминирован (word-overlap), без внешних
вызовов, всегда включён. v1 (semantic) — нет, LLM-judge, опциональный.
**Реализация:** `gates/g03_duplication.py`
**Когда гонять:** v0 — каждый push (дёшево, без API). v1 включается сам,
если judge доступен (api-ключ или `JUDGE_BACKEND=cli`) — иначе гейт
работает ровно как раньше (только v0), без деградации.

## v0 — эвристика (как раньше)
Токенизация description+body (нижний регистр, вычищены стоп-слова и слова
короче 3 символов, **без стемминга** — "хэш" и "хэши" разные токены), сравнение
с каждым скиллом в `registry_dir` по индексу Жаккара (`|A∩B| / |A∪B|`). Порог
`thresholds.duplication_similarity` (0.85). Совпадение здесь → сразу `FAIL`,
до v1 дело не доходит (без нужды тратить judge на то, что уже очевидно).

**Ограничение (решает v1):** ловит только текстуальные дубли (перефраз с
сохранением лексики/словоформ). Два скилла с одинаковой функцией, но
разными словами или даже просто другими словоформами — не поймает (нет
стемминга, значит различие в одном суффиксе уже обнуляет пересечение).

## v1 — семантическая проверка через judge (реализовано)
1. **Дешёвый lexical prefilter:** кандидат идёт к judge, только если его
   Jaccard с текущим скиллом ∈ `[duplication_semantic_prefilter,
   duplication_similarity)` — т.е. v0 уже не поймал, но и не совсем чужая
   тема (контроль стоимости на потенциально O(N²) прогоне по реестру; см.
   открытый вопрос ниже).
2. Для каждого кандидата — forced tool-use (`submit_duplication_verdict`,
   тот же паттерн, что в гейте 04/06): судья получает полный текст обоих
   скиллов, оценивает `functionally_duplicate` + `similarity` (0-1) —
   решают ли скиллы одну и ту же задачу пользователя, невзирая на слова,
   язык, структуру.
3. `FAIL`, если `functionally_duplicate=true` И `similarity >=
   thresholds.duplication_semantic_similarity` (0.8) хотя бы для одного
   кандидата.
4. Без judge — v1 молча пропускается (см. `details.note`), гейт остаётся
   на v0-результате (`PASS`, если v0 чист).

Фикстура `spreadsheet-duplicate-finder` / `dataset-redundancy-detector` —
canonical пример именно этого случая: лексический Jaccard 0.073 (v0 их не
видит), но judge уверенно определяет функциональный дубль (проверено через
`JUDGE_BACKEND=cli`, similarity=1.0).

## Вход
Путь к скиллу + `registry_dir` (по умолчанию — все остальные скиллы в
`examples/skills`, в проде — весь реестр).

## Выход
FAIL со списком `(имя_скилла, similarity)` — из v0 (`lexical_matches`) или
из v1 (`semantic_matches`, вместе с `reason` судьи в `semantic_verdicts`).

## Кэш вердиктов по паре (реализовано)
`cache_store.py` (переиспользуется гейтом 07 для токен-бюджета) — плоский
JSON-кэш `history/duplication_cache.json`, ключ `model:hash(A):hash(B)`
(хэши отсортированы — пара симметрична, `functionally_duplicate` не
зависит от того, кто A, кто B). Перед вызовом judge гейт смотрит в кэш;
при совпадении хэшей обеих сторон и той же модели — вердикт берётся из
кэша без обращения к judge. Инвалидация не по времени, а по содержимому:
изменился текст скилла → хэш другой → кэш промах → пересчёт. `model` в
ключе — вердикты разных моделей не подменяют друг друга (тот же принцип,
что `backend` в ключе истории гейта 04). `details.cache_hits` в результате
гейта показывает, сколько кандидатов не потребовали нового вызова judge.

## Известные ограничения / открытые вопросы
- **Масштаб O(N²) на холодном прогоне.** Кэш амортизирует **повторные**
  прогоны неизменного реестра (обычный случай — CI гоняет одни и те же
  тексты снова и снова), но не решает первый прогон большого реестра
  с нуля: если добавить сразу сотню новых скиллов, каждая пара всё ещё
  проходит через prefilter и, при совпадении, идёт к judge — количество
  новых пар при полном холодном прогоне по-прежнему растёт квадратично.
  Кэш снижает **предельную** стоимость (стабильный реестр почти не платит
  за повторные прогоны), не асимптотику первого прохода. Если это станет
  реальной проблемой (сотни скиллов, частые холодные прогоны) — следующий
  шаг не кэш, а снижение числа пар другим способом (например
  approximate nearest neighbor по эмбеддингам вместо полного перебора).
- **`duplication_semantic_similarity=0.8` и `duplication_semantic_prefilter=0.05`**
  — приняты как первое разумное значение, не откалиброваны на выборке
  (как и пороги гейтов 04-05).
- Настоящие эмбеддинги (cosine similarity) как альтернатива LLM-судье не
  реализованы — решили переиспользовать уже существующий `judge_client`
  (два бэкенда, api/cli) вместо добавления нового провайдера/зависимости
  (например Voyage AI) только ради гейта 03.
