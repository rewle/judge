# Роадмап — зависимости скиллов друг от друга и от внешнего состояния (design note, не реализовано)

**Пересмотрено дважды 2026-07-10.** Первая версия предполагала граф
вызовов skill→skill (`uses:` во frontmatter). Вторая версия по итогам
проверки на одном личном vault пересмотрела это на skill→file и отложила
skill→skill как неподтверждённую гипотезу. **Это было ошибкой** — `judge`
предназначен для других систем (корпоративный реестр, не этот vault), и
дизайн для него нельзя строить по выборке из одного нерепрезентативного
источника. Ниже — исправленная версия: обе гипотезы держим живыми, ни одну
не хороним по одной точке данных.

## Что реально известно (без переобобщения)
Проверены 7 скиллов в личном Obsidian vault пользователя
(`~/Projects/mind/.claude/commands/*.md`) — ни один не ссылается на другой
скилл, зато есть явные skill→file связи (`tg-post.md` читает/пишет файлы
`ТГ-канал/`, `Шаблоны/`). **Это говорит только про этот конкретный vault**,
где скиллы — по сути макросы над файловой базой знаний в Obsidian. Целевые
системы `judge` — другие (пользователь явно это указал), и там конвенция
может быть принципиально другой: multi-agent оркестрация, реестр с explicit
tool-calling между скиллами, что угодно. Пока нет данных с реальной целевой
системы, из этого факта нельзя делать вывод "skill→skill не бывает" —
можно только: "skill→file тоже встречается и его дёшево проверять,
независимо от того, бывает ли skill→skill".

## Два независимых, не взаимоисключающих направления

### A. Skill→skill (граф вызовов)
Скилл A инструктирует вызвать скилл B — либо неявно (текстом, "затем
вызови X"), либо декларативно (гипотетическое поле `uses: [skill-a]` во
frontmatter, если такая конвенция в целевой системе появится/уже есть).

**Failure mode:**
- Циклы (A→B→A) — риск зацикливания агента.
- Битая ссылка на переименованный/удалённый скилл.
- **Эскалация прав через композицию.** Скилл A с виду безобидный (`Read`),
  но инструктирует вызвать B с `Bash(*)` — эффективный blast radius шире,
  чем видно по A одному. Гейт 02 сегодня это не видит вообще.
- Несовместимость input/output между A и B по смыслу (A отдаёт CSV,
  B ждёт JSON) — ни один текущий гейт не сравнивает скиллы друг с другом
  по контракту (гейт 03 v1 близко, но там речь о дублировании функции,
  не о совместимости на стыке).
- Совокупный токен-бюджет цепочки — каждый скилл может проходить гейт 05
  по отдельности, а цепочка вместе раздута непропорционально задаче.
- Многошаговый prompt injection — инъекция входит на шаге A, проявляется
  на шаге C; однохоповые сценарии гейта 06 такое не поймают.

**Архитектура:** принципиально реестровая — нужно видеть граф целиком, не
один `skill_path`. Не вписывается в `check(skill_path)` без изменений;
нужен отдельный проход после того, как участники цепочки прошли 01-06 по
отдельности.

### B. Skill→file (общее состояние)
Скилл читает/пишет файлы внешней базы знаний — подтверждено на одном
примере (см. выше), вероятно встречается и в других системах, где скиллы
опираются на shared knowledge base/файловое хранилище, а не только там,
где проверено.

**Failure mode:**
- **Битая ссылка на файл** — скилл инструктирует "прочитай X", X
  переименован/удалён. Прямая аналогия с тем, что `vault-health` уже
  ловит для `[[wiki-ссылок]]`.
- Дрейф формата на стыке — скилл пишет файл по одному формату, другой
  потребитель (скилл или тот же скилл в будущем прогоне) ждёт другой.
- Hub-паттерн (веер, не цепочка) — несколько скиллов ссылаются на один
  общий файл; правка файла тихо задевает всех разом.

**Архитектура:** per-skill, вписывается в `check(skill_path)` без
изменений — нужен только доп. параметр `paths.external_state_root` (корень
базы знаний, не реестр скиллов). Дешевле и проще, чем A, независимо от
того, реализуем ли A вообще.

## Почему не выбирать одно направление сейчас
Оба паттерна не эксклюзивны — реальная система вполне может иметь и то, и
другое одновременно (скилл читает общий файл И вызывает другой скилл).
Дизайн `judge` должен **определять, что применимо, по факту в проверяемом
реестре**, а не по тому, что нашлось в одной выборке:
- Если в скиллах реестра встречается `uses:` (или что там окажется
  реальной конвенцией) — включать граф-проверки (A).
- Если в теле скиллов встречаются ссылки на внешние файлы — включать
  reference-check (B), независимо от того, есть ли `uses:`.
- Ни одно не должно быть жёстко закодированным единственным
  предположением — оба гейта, если оба реализовывать, должны быть
  условно активируемыми (аналог `not_configured`/`skipped` у гейтов
  04-06, когда предпосылка для проверки просто отсутствует у конкретного
  скилла/реестра).

## Набросок B (дешевле, реализовать раньше при прочих равных)
Гипотетический `gates/g07_references.py`, per-skill:
1. Извлечь упоминания путей из тела скилла вне code-блоков (тот же приём,
   что уже используют гейт 01 для путей с `\` и плейсхолдеров). Разделять
   "обязательные к существованию" (после "прочитай/read", "по шаблону из",
   "см.") и "создаваемые" (после "сохрани в/save to/запиши в" — не
   обязаны существовать заранее).
2. Резолвить относительно `paths.external_state_root` (новый параметр
   config.yaml, не совпадает с `registry_dir`). Без него — `not_configured`.
3. FAIL, если обязательная ссылка не резолвится. Детерминированно, без
   judge, можно гонять на каждый push, если `external_state_root` задан.
4. Отдельно (не гейт, а утилита-отчёт) — граф "кто на что ссылается" для
   ревьюера, меняющего общий файл.

## Набросок A (дороже, реестровый, начинать только если конвенция реальна)
- Детерминированная часть: граф по `uses:`, топологическая сортировка
  (цикл → FAIL), проверка существования каждого таргета в реестре,
  объединение `tools:` по цепочке против той же политики, что гейт 02.
- Опциональная judge-часть (переиспользовать `judge_client`, не заводить
  новый стек — тот же паттерн, что уже в гейте 03 v1): для каждого ребра
  A→B — совместим ли заявленный выход A с заявленным входом B по смыслу.
- Самая дорогая категория проверки — батч/ночной прогон по реестру, не
  PR-гейт.

## Прежде чем реализовывать что-либо из этого
Нужны данные о целевой системе (или системах), для которых `judge`
реально пишется — не с этого vault. Конкретно полезно узнать:
- Есть ли там уже механизм, которым один скилл ссылается на другой
  (explicit tool-calling, декларативное поле, что угодно)? Если да —
  какой именно синтаксис/конвенция.
- Опираются ли скиллы там на общее файловое/объектное хранилище
  (аналог этого vault) или каждый скилл самодостаточен?
Без этого любая реализация (A или B) — гадание по чужой выборке, тот же
класс ошибки, что уже был сделан в предыдущей версии этого документа.
