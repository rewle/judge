  Это меняет статус направления A с "гипотеза" на "к проектированию и

## Направление A — граф вызовов skill→skill (к реализации)

### Конвенция: поле `uses:` во frontmatter
Раз конвенцию придумываем мы — предложение, максимально простое для v1
(без спекулятивной структуры, которую негде было бы откалибровать):

```yaml
---
name: my-skill
description: ...
tools:
  - Read
uses:
  - other-skill-name
  - another-skill-name
---
```

Решения и почему:
- **Плоский список имён**, не объекты с полями `when:`/`type:` и т.п. —
  различие "условный/обязательный вызов" не влияет ни на один из проверяемых
  ниже failure mode (цикл — цикл в любом случае, эскалация прав — риск в
  любом случае), поэтому это была бы абстракция без потребителя. Если
  появится конкретный кейс, где различие нужно — добавить тогда.
- **Имя — это `frontmatter.name` целевого скилла**, не путь и не имя
  папки — тот же принцип адресации, что уже использует история гейта 04
  (`skill_name = frontmatter.get("name")`).
- **Декларация, не инференс из тела.** `uses:` — источник истины сам по
  себе, гейт не обязан подтверждать его текстом тела (симметрично тому,
  как `tools:` уже работает в гейте 02 — там тоже не проверяется, что
  каждый задекларированный инструмент реально упомянут в теле). Открытый
  вопрос ниже про обратный случай (тело зовёт скилл, не задекларированный
  в `uses:`) — это отдельный, более важный для безопасности чек.
- **Только outgoing-рёбра.** A объявляет "я использую B"; обратной связи
  "используется A" никто не объявляет — это выводится из графа, не
  авторская информация.

### Гейт 07 (`gates/g07_chain.py`, набросок готов к реализации)
**Вход:** весь `registry_dir`, не один `skill_path` — принципиально
отличается от 01-06. `run_gates.py` не умеет вызывать реестровые гейты —
нужен новый режим прогона (см. "Архитектурный сдвиг" ниже).

**Детерминированные проверки (v1, без judge):**
1. **Существование ссылки.** Каждое имя в `uses:` должно резолвиться в
   реальный скилл реестра (по `frontmatter.name`). Иначе — `FAIL`
   "`uses:` ссылается на несуществующий скилл 'X'".
2. **Ацикличность.** Топологическая сортировка по всему графу реестра
   (не только скиллам с `uses:` — любой скилл может быть узлом). Цикл →
   `FAIL` с именем цикла (`A → B → A`).
3. **Эскалация прав через композицию.** Для каждого скилла — транзитивное
   замыкание `uses:` (полный граф зависимостей, не только прямые), 
   объединение `tools:` по всем узлам замыкания, прогон объединения через
   ту же политику, что гейт 02 (`tools_allowlist.yaml` — вынести общую
   логику из `gates/g02_permissions.py` в переиспользуемую функцию, не
   дублировать). `FAIL`/флаг-на-ревью, если замыкание тянет
   `always_flag_for_review`-инструмент, которого нет в собственном
   `tools:` скилла — ровно кейс "A с виду безобидный, но через B получает
   `Bash(*)`".

**Отложено за пределы v1 (см. failure mode ниже, зачем они всё ещё в
списке, но не в первой реализации):**
- Совместимость input/output между A и B по смыслу — потребовала бы ещё
  одной новой конвенции (декларация "что скилл производит/ожидает"),
  которой пока тоже нет; не изобретать вторую конвенцию заодно с первой.
- Совокупный токен-бюджет цепочки, многошаговый prompt injection — оба
  требуют behavioral-инфраструктуры на несколько скиллов сразу (в духе
  гейта 05/06, но для графа) — дорого, отдельный этап после того, как
  статическая часть (существование/циклы/права) заработает и покажет, что
  конвенция `uses:` вообще используется осмысленно.

### Открытый вопрос — обратная проверка
Более важный для безопасности случай, чем "задекларировал, но не
использует": **тело скилла инструктирует вызвать скилл, не
задекларированный в `uses:`** — необъявленная зависимость, гейт 02 её не
увидит вообще (он даже не знает, что нужно искать). Технически это скан
тела на упоминания имён других скиллов реестра (похожий приём на то, что
уже делает гейт 03 для сравнения текстов) — вне code-спанов, с некоторой
эвристикой на "это призыв вызвать скилл", а не случайное упоминание имени
в другом контексте. Не спроектировано на уровне регэкспа/эвристики —
следующий шаг детализации, если направление A подтверждается в реализации.

### Архитектурный сдвиг (по-прежнему актуален)
Все гейты 01-06 — `check(skill_path) -> GateResult`, вызываются
`run_gates.py` по одной папке. Гейт 07 принципиально реестровый — нужна
сигнатура вида `check_registry(registry_dir) -> dict[skill_name, GateResult]`
и новый режим в `run_gates.py` (например `--check-chains`), который
запускается **после** того, как все скиллы реестра прошли 01-06 по
отдельности (нет смысла проверять граф, если его узлы сами по себе не
прошли базовые гейты).

## Направление B — skill→file (дешёвое дополнение, не приоритет)
Из проверки на личном vault: скиллы читают/пишут файлы общей базы знаний
(`tg-post.md` → `ТГ-канал/Стиль и контекст.md` и т.п.). Не опровергнуто и
не обязательно применимо к целевым системам (см. историю ревизий выше) —
но раз это per-skill и дёшево (`check(skill_path)`, без реестрового
прохода, без judge), можно сделать независимо от того, приоритетно ли это:
не мешает A, не требует того же архитектурного сдвига.

**Набросок (не детализирован дальше этого уровня, т.к. не приоритет):**
извлечь пути после "прочитай/read"/"по шаблону из" вне code-блоков,
резолвить относительно нового `paths.external_state_root`, `FAIL` если не
резолвится. Подробности — если/когда дойдут руки после A.
