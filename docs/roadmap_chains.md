# Роадмап — внешние зависимости скиллов (design note, не реализовано)

**Пересмотрено 2026-07-10.** Первая версия этого документа предполагала, что
"цепочка" — это граф вызовов skill→skill (скилл A инструктирует вызвать
скилл B). Проверка на реальных данных это не подтвердила — см. ниже. Модель
пересмотрена на skill→file: скиллы связаны не друг с другом, а через общее
состояние (файлы базы знаний, которую они читают и пишут).

## Эмпирическая проверка (что на самом деле происходит)
Проверены все 7 реально используемых скиллов в пользовательском Obsidian
vault (`~/Projects/mind/.claude/commands/*.md` — там же, где живёт judge).
**Ни один скилл не упоминает другой скилл в своём теле.** Zero
skill-to-skill ссылок. Зато, например, `tg-post.md` явно инструктирует:

```
Шаг 2. Прочитай `ТГ-канал/Опубликованные посты.md`.
Шаг 3. Прочитай `ТГ-канал/Стиль и контекст.md`.
Шаг 6. Сохрани в `ТГ-канал/Черновики/YYYY-MM-DD [тема].md`
       по шаблону из `Шаблоны/Черновик ТГ-поста.md`.
```

Связь есть — но это "скилл → файл базы знаний", не "скилл → скилл". Выборка
маленькая (один личный vault, не корпоративный реестр), но она единственная
реальная, которая была под рукой, и она однозначна: там, где скиллы
реально пишутся и используются, авторы связывают их через общий стейт, а
не через явные вызовы друг друга. Формальная декларативная конвенция вида
`uses: [skill-a, skill-b]` (кандидат A из первой версии этого документа)
не только не существует — ей, похоже, и не откуда взяться: агент выбирает
скилл по смыслу задачи каждый раз заново, а не по объявленной автором связи
с другим скиллом.

## Что это меняет
Прежние failure mode (циклы вызовов, эскалация прав через композицию скиллов,
многошаговый prompt injection через цепочку вызовов) остаются валидными
*гипотезами на будущее*, но привязаны к паттерну, которого сейчас нет в
наблюдаемых данных — реализовывать их сейчас значит гейтить против
несуществующей угрозы. Основной риск на практике другой и гораздо более
приземлённый:

## Failure mode, которые актуальны уже сейчас
- **Битая ссылка (главный кейс).** Скилл инструктирует "прочитай X" —
  X переименован, перемещён или удалён из базы знаний. Скилл продолжает
  формально проходить все гейты 01-06 (они видят только текст самого
  скилла), но на практике сломан: агент попытается прочитать
  несуществующий файл и либо провалит задачу, либо начнёт импровизировать
  вместо реального содержимого (см. groundedness в гейте 04 — тот же
  класс проблемы, но триггер другой). Прямая аналогия: `vault-health`
  уже ловит битые `[[wiki-ссылки]]` внутри заметок vault'а — это тот же
  чек, только источник ссылок не заметка, а скилл.
- **Дрейф формата на стыке.** Скилл A на шаге 6 сохраняет файл по
  какому-то формату/пути; если формат неявно меняется (другая структура
  шаблона, другое имя поля), а другой скилл (или тот же скилл в будущем
  прогоне) ожидает старый формат — рассинхрон, который ни один текущий
  гейт не увидит, т.к. каждый скилл проверяется независимо от состояния
  файлов, которые он трогает.
- **Скрытый hub-паттерн (не цепочка, а веер).** Несколько скиллов могут
  ссылаться на один и тот же файл (например `Шаблоны/*` или общую
  памятку) — правка этого файла тихо задевает сразу несколько скиллов.
  Это не проверка одного скилла, а вопрос "если я поменяю файл X, что
  сломается" — годится не как гейт, а как отчёт/утилита для ревьюера.

## Что откладывается (пока нет оснований делать сейчас)
- Граф зависимостей `uses:`, проверка циклов, объединение прав `tools:`
  по цепочке вызовов, многошаговые redteam-сценарии через несколько
  скиллов подряд. Если в реестре реально появится skill-to-skill
  делегирование (не факт, что появится — см. эмпирику выше), к этому
  разделу можно будет вернуться; пока это гипотеза без подтверждения.

## Набросок гейта (реалистичный масштаб, дешевле прежнего плана)
В отличие от первой версии, это **не обязательно реестровая проверка** —
"есть ли файл X, на который ссылается скилл" проверяется по одному скиллу,
не требует видеть весь реестр разом. Вписывается в существующую модель
`check(skill_path) -> GateResult`, гипотетически как `gates/g07_references.py`:

1. **Извлечь упоминания путей** из тела скилла (вне code-блоков/спанов —
   тот же приём, что уже используют гейт 01 для путей с `\` и плейхолдеров).
   Триггерные глаголы по-русски/английски: "прочитай/read" и
   "по шаблону из/см." для обязательных ссылок; "сохрани в/save to/запиши в"
   для ссылок, которые не обязаны существовать заранее (файл создаётся
   впервые) — их не проверять на существование, но можно логировать для
   hub-отчёта.
2. **Резолвить относительно корня базы знаний**, которую скиллы читают —
   это **не** `registry_dir` (реестр самих скиллов), а отдельный путь,
   config.yaml нужно расширить новым `paths.external_state_root` (для
   этого репозитория, если тестировать на примере — путь к
   `~/Projects/mind`). Без этого параметра гейт не может знать, откуда
   резолвить пути → `not_configured`, не блокирует цепочку.
3. **FAIL**, если хотя бы один "обязательный к существованию" путь не
   резолвится. Полностью детерминированно, без judge — дешевле почти
   всех остальных гейтов, можно гонять на каждый push (как 01-02), если
   `external_state_root` задан.
4. Hub-отчёт (кто на что ссылается) — отдельная утилита, не гейт: годится
   для ревьюера, который меняет общий файл и хочет знать зону поражения.

## Прежде чем реализовывать
Нужна вторая точка данных — сегодняшняя проверка это один личный vault на
7 скиллов. Если/когда появится доступ к реальному корпоративному реестру
(тот, ради которого вообще затевался `judge`), стоит повторить ту же
проверку там: тоже ли skill→file, или в более формальной корпоративной
среде всё-таки возникает skill→skill. Это дешёвая проверка (тот же grep,
что был сделан здесь) — имеет смысл сделать её первым шагом, а не
писать гейт вслепую под одну выборку.
