## Направление A — граф вызовов skill→skill (к реализации)

### Конвенция: поле `uses:` во frontmatter
Раз конвенцию придумываем мы — предложение, максимально простое для v1
(без спекулятивной структуры, которую негде было бы откалибровать):

```yaml
---
name: my-skill
description: ...
tools:
  - Read
uses:
  - other-skill-name
  - another-skill-name
---
```

Решения и почему:
- **Плоский список имён**, не объекты с полями `when:`/`type:` и т.п. —
  различие "условный/обязательный вызов" не влияет ни на один из проверяемых
  ниже failure mode (цикл — цикл в любом случае, эскалация прав — риск в
  любом случае), поэтому это была бы абстракция без потребителя. Если
  появится конкретный кейс, где различие нужно — добавить тогда.
- **Имя — это `frontmatter.name` целевого скилла**, не путь и не имя
  папки — тот же принцип адресации, что уже использует история гейта 04
  (`skill_name = frontmatter.get("name")`).
- **Декларация, не инференс из тела.** `uses:` — источник истины сам по
  себе, гейт не обязан подтверждать его текстом тела (симметрично тому,
  как `tools:` уже работает в гейте 02 — там тоже не проверяется, что
  каждый задекларированный инструмент реально упомянут в теле). Открытый
  вопрос ниже про обратный случай (тело зовёт скилл, не задекларированный
  в `uses:`) — это отдельный, более важный для безопасности чек.
- **Только outgoing-рёбра.** A объявляет "я использую B"; обратной связи
  "используется A" никто не объявляет — это выводится из графа, не
  авторская информация.

### Гейт 07 (`gates/g07_chain.py`, реализовано 2026-07-10)
**Вход:** весь `registry_dir`, не один `skill_path` — принципиально
отличается от 01-06. `run_gates.py --check-chains [--registry <путь>]`
(по умолчанию — `paths.registry_dir` из `config.yaml`).

**Детерминированные проверки (v1, без judge) — все реализованы:**
1. **Существование ссылки.** Каждое имя в `uses:` должно резолвиться в
   реальный скилл реестра (по `frontmatter.name`). Иначе — `FAIL`
   "`uses:` ссылается на несуществующий скилл 'X'". Фикстура:
   `examples/chains/chain-dangling`.
2. **Ацикличность.** DFS с раскраской по всему графу реестра (не только
   скиллам с `uses:` — любой скилл может быть узлом). Цикл → `FAIL` с
   путём цикла (`A → B → A`). Фикстуры: `chain-cycle-a`/`chain-cycle-b`.
3. **Эскалация прав через композицию.** Для каждого скилла — транзитивное
   замыкание `uses:` (полный граф, не только прямые рёбра, cycle-safe через
   visited-set), объединение `tools:` по замыканию, прогон разницы
   (замыкание минус собственные tools:) через `evaluate_tools()` —
   вынесенную из `gates/g02_permissions.py` общую функцию политики (та же
   логика allowlist/requires_justification/always_flag_for_review, без
   дублирования). Инструменты, унаследованные транзитивно, оцениваются с
   `justification=""` — обоснование, написанное для собственных tools:,
   не покрывает унаследованные автоматически. Фикстуры:
   `chain-escalation-narrow` (получает `Bash(*)` от `chain-escalation-wide`,
   FAIL) vs `chain-orchestrator`/`chain-worker` (оба только `Read`, PASS).

4. **Обратная проверка.** Более важный для безопасности случай, чем
   "задекларировал, но не использует": тело скилла упоминает вызов
   скилла, не задекларированного в `uses:` — необъявленная зависимость,
   гейт 02 её вообще не видит (не знает, что искать). Реализовано как
   узкая эвристика: `_mentioned_skill_names()` ищет точное совпадение
   имени скилла реестра внутри backtick-спана (`` `skill-name` ``) вне
   fenced-блоков — минимизирует ложные срабатывания ценой пропуска
   вызовов, упомянутых без backtick-оформления (например просто "вызови
   skill-name" без кавычек не поймается). Не различает "инструктирует
   вызвать" от "упоминает по другой причине" (например `` "не используй
   `X`" `` — отрицание не распознаётся). Фикстура:
   `chain-undeclared-caller` (упоминает `` `chain-worker` `` в теле, не
   декларируя в `uses:` — FAIL).

Все 5 сценариев (существование/цикл/dangling/эскалация/необъявленный
вызов) проверены `python3 run_gates.py --check-chains --registry
examples/chains` — совпадает с ожиданиями по каждой фикстуре.

**Отложено:**
- Совместимость input/output между A и B по смыслу — **реализовано
  2026-07-11, см. следующий раздел.**
- Совокупный токен-бюджет цепочки, многошаговый prompt injection — оба
  требуют behavioral-инфраструктуры на несколько скиллов сразу (в духе
  гейта 05/06, но для графа) — дорого, отдельный этап, если/когда
  конвенция `uses:` покажет, что реально используется на практике.

## I/O-совместимость между скиллами (реализовано 2026-07-11)

### Модель: не типы, а естественноязыковой контракт
Скилл в этой экосистеме не функция со строгой сигнатурой — агент читает
`SKILL.md` скилла B, выполняет инструкции, результат ложится в контекст
разговора, которым продолжает пользоваться A. "Совместимость" здесь не
"тип на входе = тип на выходе", а "то, что B заявляет как результат,
покрывает то, что A, судя по его тексту, ожидает получить, вызывая B".
Формальная типизация (JSON Schema и т.п.) переоценила бы строгость
контракта, которого в этой модели физически нет — соответственно, как и
`description:`, проверка должна быть judge-based, не парсингом.

### Конвенция: одно новое поле `provides:`
```yaml
---
name: chain-worker
description: Читает CSV-файл и считает сумму по числовой колонке. Использовать когда нужен быстрый подсчёт суммы без открытия таблицы целиком.
tools:
  - Read
provides: Число — сумма значений указанной колонки CSV.
---
```

Решения и почему:
- **Только `provides:`, без `expects:`.** Симметричная пара
  provides/expects (по аналогии с Debian `Provides:`/`Depends:`) была бы
  логичнее, но добавила бы вторую новую конвенцию сразу после первой
  (`uses:`) — тот же класс избыточности, что уже отклонён для `uses:`
  (`when:`/`type:`). "Что скилл A ожидает получить, вызывая B" и так
  видно из существующего текста A (`description:` + тело, там же, где
  разбирает `uses:` обратная проверка) — судья читает это, а не парсит
  отдельное поле. Одно новое поле вместо двух.
- **Свободный текст**, не структура — см. модель выше: это не тип, а
  формулировка результата, симметрично `description:` по тону и формату.
- **Опционально.** Скилл без `provides:` продолжает работать как раньше;
  для рёбер `uses:`, где у B нет `provides:`, проверка этого ребра просто
  пропускается (не `FAIL` за отсутствие поля — конвенция новая, внедрение
  постепенное, тот же принцип, что у `eval.yaml` в гейте 05).
- **Название.** `provides:` выбрано (не `output:`/`returns:`) как парная
  метафора к `uses:` — B, которого используют, что-то предоставляет.

### Проверка (расширение гейта 07, не новый гейт 08) — реализовано
Переиспользует `judge_client` (тот же паттерн, что гейт 03 v1:
детерминированная часть гейта работает всегда, judge-часть — опционально
поверх).

Для каждого ребра `A → B` в графе `uses:`, где у B есть непустой
`provides:`:
1. Forced tool-use (`submit_io_compat_verdict`: `compatible: bool`,
   `score: 0..1`, `reason: str`) — судья получает полный текст A
   (`_read_skill` → `text`, весь SKILL.md) и `description`/`provides:` B.
2. `FAIL`, если `compatible=false` ИЛИ `score < thresholds.chain_io_compat_min_score`
   (0.6 — первое разумное значение, не откалибровано, как и остальные
   пороги 03-05).
3. Без judge (`JudgeNotConfigured`) — весь I/O-слой пропускается
   gate-wide (проверено на реальном прогоне без ключа: пара
   `chain-io-mismatch-*`, несовместимая при judge, при его отсутствии —
   `PASS`, как ожидалось).
4. Рёбра, где у B нет `provides:` — пропускаются поштучно (не всей
   проверкой) — цикл просто `continue`, без пометки в `details` (открытый
   пункт ниже).

### Стоимость
Ограничена количеством рёбер `uses:` с обеих сторон (валидное ребро +
непустой `provides:` у B) — на практике сильно меньше, чем O(N²) у
гейта 03 (там сравниваются все пары реестра, здесь только объявленные
зависимости). Не нужен lexical prefilter, как у гейта 03 v1.

### Фикстуры
`chain-worker` получил `provides:` — уже существующий `chain-orchestrator`
(вызывает его явно за суммой) стал первым `PASS`-примером совместимости.
Новая пара `chain-io-mismatch-consumer`/`chain-io-mismatch-provider`:
consumer по тексту явно ожидает числовую сумму расходов в рублях,
provider заявляет `provides:` HTML-страницу результатов поиска без
числовых значений — `FAIL`. Проверено через `JUDGE_BACKEND=cli`:
score=0.03, судья дал развёрнутое обоснование несовместимости
("смысловое назначение B никак не пересекается с ожиданием A").

### Известные ограничения
- **Нет multi-run стабильности** (в отличие от гейта 04 с 8 прогонами и
  sigma) — один прогон на ребро, контроль стоимости важнее стабильности
  здесь; осознанный компромисс, не откалиброван.
- **`chain_io_compat_min_score=0.6`** не откалиброван.
- Качество проверки зависит от того, насколько содержательно описан
  `provides:` — расплывчатая формулировка снизит уверенность судьи, гейт
  это отдельно не проверяет (не дублирует description_quality-подобные
  критерии рубрики гейта 04 для нового поля).
- Рёбра без `provides:` у B пропускаются молча (без пометки в `details`,
  в отличие от `not_configured`-статусов у других гейтов) — не видно из
  отчёта, что часть рёбер вообще не проверена на совместимость; можно
  доработать позже, если это окажется реальной проблемой на практике.
- Не проверено настоящим api-бэкендом (только `JUDGE_BACKEND=cli`, как и
  большинство judge-частей других гейтов на момент реализации).

### Архитектурный сдвиг и включение в CI-порядок (реализовано)
Все гейты 01-06 — `check(skill_path) -> GateResult`, вызываются
`run_gates.py` по одной папке. Гейт 07 принципиально реестровый —
`check_registry(registry_dir) -> dict[skill_name, GateResult]`.

`run_gates.py` (без `--skill`/`--path`, т.е. полный прогон реестра)
теперь **автоматически** гоняет гейт 07 после того, как все скиллы
реестра прошли 01-06 — если хоть один скилл упал на 01-06, гейт 07
пропускается с явным сообщением ("нет смысла проверять граф, если его
узлы сами по себе не прошли базовые гейты"), не запускается вслепую.
`--skip-chains` отключает это для точечной отладки. При проверке одного
скилла (`--skill`/`--path`) гейт 07 не запускается — гонять граф целиком
ради одного скилла бессмысленно. `--check-chains` по-прежнему доступен
для прогона только гейта 07, отдельно от 01-06 (быстрая итерация на
графе без ожидания per-skill гейтов).

## Направление B — skill→file (дешёвое дополнение, не приоритет)
Из проверки на личном vault: скиллы читают/пишут файлы общей базы знаний
(`tg-post.md` → `ТГ-канал/Стиль и контекст.md` и т.п.). Не опровергнуто и
не обязательно применимо к целевым системам (см. историю ревизий выше) —
но раз это per-skill и дёшево (`check(skill_path)`, без реестрового
прохода, без judge), можно сделать независимо от того, приоритетно ли это:
не мешает A, не требует того же архитектурного сдвига.

**Набросок (не детализирован дальше этого уровня, т.к. не приоритет):**
извлечь пути после "прочитай/read"/"по шаблону из" вне code-блоков,
резолвить относительно нового `paths.external_state_root`, `FAIL` если не
резолвится. Подробности — если/когда дойдут руки после A.
