# Роадмап — цепочки скиллов (design note, не реализовано)

Все гейты 01-06 проверяют **один скилл в изоляции**. По мере роста реестра
скиллы начинают неявно опираться друг на друга: пост в ТГ-канал берёт
контекст из карточки контакта, дайджест новостей может стать источником
для поста, один скилл прямо в тексте инструктирует "после этого вызови
скилл X". Сегодня это работает только за счёт того, что модель-агент сама
решает, какой скилл вызвать дальше, по смыслу задачи — никакой явной
структуры "это цепочка" в реестре нет.

## Что такое "цепочка" здесь (открытый вопрос №1)
Два кандидата, и неочевидно, какой ближе к реальности платформы:

**A. Декларативная.** Новое опциональное поле `uses: [skill-a, skill-b]` во
frontmatter — автор явно заявляет, что скилл рассчитывает на вызов других
скиллов реестра как часть своей работы. Машинно проверяемо статически (граф
зависимостей), но требует, чтобы такое поле реально появилось в конвенции
авторинга скиллов — сейчас его нет, и не факт, что появится: агент сегодня
выбирает скиллы по описанию/триггеру динамически, а не по декларации автора.

**B. Поведенческая.** Никакого нового поля не вводим. Вместо статического
графа — behavioral-прогон (в духе гейта 05): дать агенту составную
пользовательскую задачу, которая по смыслу должна вызвать несколько
скиллов реестра подряд, прогнать и посмотреть, что агент реально выбрал,
в каком порядке, и проверить ниже перечисленные failure mode эмпирически.

Дешевле начать с B (не требует менять конвенцию авторинга скиллов, которую
не мы контролируем), но B даёт только выборочное покрытие (какие сценарии
прогнали, те и проверили), тогда как A потенциально даёт полное покрытие
графа зависимостей, если поле `uses:` вообще приживётся.

## Failure mode, которые не ловит ни один текущий гейт
- **Циклы.** A вызывает B, B вызывает A — риск зацикливания агента.
- **Битая ссылка.** A ожидает существования B (по имени/описанию), B
  переименован или удалён из реестра — тихая деградация, а не явный fail.
- **Эскалация прав через композицию.** Гейт 02 проверяет `tools:` одного
  скилла. Скилл A с виду безобидный (`Read`) может внутри текста
  инструктировать вызвать скилл B с `Bash(*)` — эффективный blast radius
  цепочки шире, чем то, что видно по любому отдельному скиллу. Гейт 02 в
  сегодняшнем виде такое не увидит вообще.
- **Несовместимость input/output.** A производит CSV с колонками X,Y,Z,
  B ожидает JSON с полем `foo` — по отдельности оба скилла могут быть
  качественными (пройти гейт 04), но вместе не работают. Ни один текущий
  гейт не сравнивает скиллы друг с другом по контракту (кроме гейта 03,
  но там речь о дублировании функции, не о совместимости на стыке).
- **Совокупный токен-бюджет.** Каждый скилл цепочки может по отдельности
  проходить гейт 05, а вся цепочка вместе — раздута непропорционально
  задаче. Гейт 05 меряет один скилл, не сумму по цепочке.
- **Многошаговый prompt injection.** Гейт 06 проверяет устойчивость одного
  скилла к инъекции в его собственном контексте. В цепочке инъекция может
  войти на шаге A (через внешний контент) и проявиться только на шаге C,
  через несколько шагов — сценарии гейта 06 однохоповые, такое не поймают.

## Архитектурный сдвиг
Все текущие гейты — `check(skill_path) -> GateResult`, работают на одном
скилле, вызываются `run_gates.py` по одной папке за раз. Проверка цепочки
принципиально **реестровая**: нужно видеть несколько скиллов и связи между
ними одновременно, а не один `skill_path`. Это не вписывается в текущий
цикл `run_gates.py` (перебор папок one-by-one, fail-fast на уровне одного
скилла) — нужен отдельный проход после того, как все скиллы цепочки
самостоятельно прошли 01-06 (нет смысла проверять совместимость того, что
по отдельности уже сломано).

## Набросок (не реализовано)
Если пойти по пути A (декларативный `uses:`) — гипотетический гейт 07:
- **Детерминированная часть:** построить граф зависимостей по `uses:` во
  всём реестре; топологическая сортировка → FAIL с именем цикла, если граф
  не ациклический; проверить, что каждый `uses:`-таргет существует в
  реестре; посчитать объединение `tools:` по всем узлам цепочки и прогнать
  через ту же политику `tools_allowlist.yaml`, что гейт 02 — здесь и
  всплывёт "невинный A эскалирует права через B".
- **Опциональная judge-часть** (тот же паттерн, что уже есть у гейта 03 v1
  и 04 — переиспользовать `judge_client`, не заводить третий стек): для
  каждого ребра A→B — совместим ли заявленный выход A с заявленным входом
  B по смыслу (forced tool-use verdict, как `submit_duplication_verdict`).

Если пойти по пути B (поведенческий) — расширение `behavioral.py`:
составные сценарии на несколько скиллов сразу, дороже и реже гонять (по
аналогии с гейтом 05 — это уже недешёвый паттерн), скорее батчем/ночным
прогоном, чем на каждый PR.

## Стоимость и место в пайплайне
Самая дорогая категория проверки из всех: либо O(N²) на рёбра графа с
LLM-судьёй на каждое (путь A), либо полноценные многошаговые agent-прогоны
через несколько реальных скиллов (путь B). Ни то, ни другое не годится на
каждый push — это кандидат на редкий батч/ночной прогон по всему реестру,
не на PR-гейт для одного скилла.

## Прежде чем реализовывать
Нужно сначала понять: приживётся ли `uses:` как реальная конвенция
авторинга в этом реестре скиллов, или цепочки так и останутся неявными
(агент решает сам, без деклараций автора). Если конвенции не будет — путь A
не имеет смысла реализовывать, и роадмап сводится к пути B.
