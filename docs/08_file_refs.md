# Гейт 08 — ссылки на внешние файлы (skill→file)

**Детерминизм:** полный, без judge, без внешних вызовов.
**Реализация:** `gates/g08_file_refs.py`
**Когда гонять:** каждый push, дёшево. Позиционирован в `config.yaml`
сразу после `02_permissions` — cheap-детерминированные гейты раньше
judge-зависимых (03+), номер файла (08) — по порядку появления в проекте
(симметрично гейту 07), не по позиции в последовательности прогона.

## Контекст: направление B роадмапа цепочек
См. `docs/roadmap_chains.md`. В отличие от гейта 07 (граф skill→skill
через `uses:`), это skill→file: скилл читает/пишет файлы общей базы
знаний вне реестра скиллов (шаблон, конфиг, датасет) — не другой скилл,
поэтому гейт 07 такую зависимость не видит и видеть не должен (у неё нет
`uses:`-имени, чтобы резолвить).

## Вход
Тело SKILL.md (вне fenced code-блоков) + `paths.external_state_root` из
`config.yaml` (опционально).

## Логика
1. Без `paths.external_state_root` в конфиге — `SKIPPED` целиком, gate
   выключен по умолчанию: конвенция новая, внедрение постепенное (тот же
   принцип, что `eval.yaml` у гейта 05 или `provides:` у гейта 07).
2. С заданным корнем: regex ищет путь к файлу сразу после триггерной фразы
   (`прочитай`/`прочти`/`read`/`по шаблону из`/`template from`/`из файла`/
   `from file`), токен должен заканчиваться расширением файла
   (`\.[A-Za-z0-9]{1,5}`), опционально в backtick-спане.
3. Каждый найденный путь резолвится относительно `external_state_root`.
   Не резолвится → `FAIL` со списком отсутствующих путей.
4. Ссылок не найдено → `PASS` (не ошибка — не каждый скилл обязан читать
   внешние файлы).

## Почему узкая эвристика
Тот же выбор, что у `_mentioned_skill_names` в гейте 07 (обратная проверка
`uses:`): требование триггерной фразы И расширения файла в токене
минимизирует ложные срабатывания ценой пропуска путей без явного
расширения или упомянутых без триггерного слова рядом (например, просто
"см. также `some-config`" без "прочитай" не поймается). Разработчик,
уверенный, что путь должен проверяться, а не проверяется — известное и
принятое ограничение, не баг.

## Фикстуры
`examples/external_state/templates/onboarding-template.md` — корень задан
в примере как `examples/external_state`.
- `examples/skills/file-ref-valid-skill` — "Прочитай
  `templates/onboarding-template.md`" → файл существует → `PASS`.
- `examples/skills/file-ref-broken-skill` — "Прочитай
  `templates/offboarding-template.md`" → файла нет → `FAIL`.

По умолчанию в `config.yaml` `external_state_root` не задан, поэтому при
обычном прогоне реестра обе фикстуры дают `SKIPPED`, не `PASS`/`FAIL` —
чтобы увидеть логику в действии, нужно явно передать
`external_state_root=examples/external_state` в `check()` (см. проверку
в `docs/roadmap_chains.md`/историю сессии) или задать путь в конфиге.

## Известные ограничения
- Эвристика не различает "инструктирует прочитать" от упоминания пути по
  другой причине (например в отрицании "не читай X") — то же
  задокументированное ограничение, что у обратной проверки гейта 07.
- Не проверяет **содержимое** файла — только факт существования по пути.
  Смысловая проверка (соответствует ли файл ожиданиям скилла) — не задача
  этого гейта, для skill↔skill есть аналог в `provides:`/I/O-совместимости
  гейта 07, для skill→file аналога сознательно нет (не обязательно
  применимо, см. `roadmap_chains.md`).
- Не тестировалось на реальной целевой системе — только на синтетических
  фикстурах, как и остальной проект на момент реализации.
