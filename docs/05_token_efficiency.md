# Гейт 05 — token efficiency (behavioral)

**Детерминизм:** нет — зависит от прогона модели.
**Статус:** реализован (`gates/g05_token_efficiency.py`), требует ключ
(см. README) и `eval.yaml` рядом со SKILL.md — без него `skipped`.
**Когда гонять:** вместе с гейтом 04 (тот же дорогой прогон).

## Вход
`examples/skills/<name>/eval.yaml`:
```yaml
query: "..."                 # задача для агента
input: "..."                 # опционально: данные, с которыми работает задача
expected_behavior:
  - "..."
  - "..."
agent_system_prompt: "..."   # опционально, дефолт behavioral.DEFAULT_AGENT_SYS
```

## Логика
1. **Дёшево, без генерации:** `messages.count_tokens` (реальный endpoint
   Anthropic API, без инференса) считает входные токены для baseline
   (без скилла) и with-skill (SKILL.md в контексте). `overhead_pct = (with -
   baseline) / baseline * 100`.
2. Если `overhead_pct <= thresholds.token_efficiency_max_overhead_pct` (30%)
   — сразу PASS, дальше не тратимся: overhead приемлем независимо от того,
   что он даёт.
3. Если выше — обязаны обосновать приростом coverage: два реальных прогона
   агента (baseline/with-skill) + judge-оценка покрытия
   `expected_behavior` (та же forced tool-use схема, что в гейте 04, не
   regex). `gain_pp = (with_skill_coverage - baseline_coverage) * 100`.
4. FAIL если `gain_pp < thresholds.token_efficiency_min_coverage_gain_pp`
   (10 п.п.) — overhead есть, а пользы от него практически нет.

Именно так задумана фикстура `token-bloated-skill`: тривиальная задача
(переименовать 3 колонки CSV), модель и без скилла справится полностью —
`gain_pp` должен выйти около нуля, а тело скилла явно раздуто → FAIL.

## Почему не «просто сравнить абсолютные токены»
Абсолютный overhead ничего не говорит сам по себе — скилл на сложную задачу
законно может стоить много токенов, если он реально поднимает coverage.
Порог в 30% — это не «лимит на скилл», а точка, после которой мы обязаны
спросить пруф пользы, а не верить на слово.

## Известные ограничения / открытые вопросы
- **Порог `min_coverage_gain_pp=10`** — принят как первое разумное значение,
  не откалиброван на выборке; вероятно потребует подстройки, когда появятся
  реальные прогоны.
- **Не у каждого скилла есть `eval.yaml`.** Для CI в реальном реестре это,
  вероятно, стоит сделать обязательным полем PR (авторская задача сама
  описывает свой поведенческий тест), а не опциональным файлом.
- Латентность прогона в дизайн не включена (только токены) — если станет
  важна, добавить отдельным полем в `details`, а не смешивать с overhead_pct.
- Не проверено живым ключом — на момент реализации ключа не было
  (см. память `project-judge-skill-ci`).
