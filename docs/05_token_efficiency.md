# Гейт 05 — token efficiency (behavioral)

**Детерминизм:** нет — зависит от прогона модели.
**Статус:** не реализован (`gates/g05_token_efficiency.py` — заглушка).
**Когда гонять:** вместе с гейтом 04 (тот же дорогой прогон behavioral-теста).

## Логика (план)
1. Взять поведенческий тест скилла (задача + ожидаемые пункты покрытия —
   как в слое 3 существующей методологии).
2. Прогнать дважды: (a) baseline — модель без скилла в контексте,
   (b) with-skill — модель со скиллом.
3. Сравнить: coverage(a) vs coverage(b), токены контекста(a) vs (b),
   латентность(a) vs (b).
4. FAIL если прирост токенов (`(b-a)/a` в процентах) превышает
   `thresholds.token_efficiency_max_overhead_pct` (30%), а прирост coverage
   не пропорционален — т.е. скилл платит непропорционально дорого за
   покрытие, которое даёт.

Именно этот сценарий у фикстуры `token-bloated-skill`: coverage(a) ≈
coverage(b), но токены(b) намного больше — задача тривиальна, а текст скилла
раздут нерелевантными рассуждениями.

## Открытые вопросы для реализации
- "Пропорционально" нужно формализовать — простой линейный порог

- Считать ли только токены самого SKILL.md в контексте, или полную стоимость
  прогона (включая tool calls, вызванные скиллом)
