# TODO: сервис-эмулятор пользователя по декларативному сценарию

Предложение Ивана Соколова (15.07.2026): по декларативному описанию сценария
тестирования эмулировать пользователя в диалоге со «стендом» (внешней
системой под тестом) через её ручки, снять трейс, проверить его двумя
способами — точно (например, «функция X была вызвана») и семантически
(judge). План согласован с Александром 2026-07-15, статус —
**реализовано, Фазы 0-4 готовы и проверены живыми прогонами** (см. фазы
ниже). Открытое допущение про сращивание с графом `uses:`/chains/groups —
осталось открытым, см. соответствующий раздел ниже.

## Подтверждённые решения

1. **Сценарий** — свободная форма (без жёсткой схемы): текстовое описание
   цели диалога и того, что нужно проверить — часть точно/императивно,
   часть семантически.
2. **Стенд** — любой внешний сервис, подключается по общему HTTP-контракту
   (минимальный набор хендлов: открыть сессию / отправить сообщение /
   получить состояние-трейс).
3. **Первый стенд для обкатки — фейковый эхо-стенд** внутри этого репо,
   держит сессию в фоне, пересылает сообщения. Нужен, чтобы обкатать сам
   механизм эмуляции до подключения реальных внешних сервисов.
4. **Ядро эмулятора** — отдельный клиент/обвязка НАД текущим cli-бэкендом
   (`claude -p`): историю диалога ведём сами вне `judge_client.py`, на
   каждый вызов передаём накопленный контекст целиком. Api-бэкенд не
   требуется.
5. **Исполнение сценария сандбоксировано на уровне процесса эмулятора**:
   раннер работает в изолированном окружении, сеть — только к заявленному
   стенду (плюс то, что нужно самой cli-обвязке).
6. **Трейс** — append-only лог событий, которые эмулятор сам пишет по ходу
   прогона (сообщения, вызовы к стенду, аргументы, ответы) — не парсинг
   текста постфактум.
7. **Проверка** — двухчастная: точная сверка трейса (assert без LLM) +
   семантическая оценка через существующий `judge_client` (по аналогии с
   `_judge_transcript` из `gates/g06_redteam.py`).
8. **Размещение** — отдельный инструмент/команда рядом с `run_gates.py`, не
   в основном pipeline гейтов.

## Фазы

- **Фаза 0** — контракт-документ по конвенции проекта
  (`docs/09_scenario_emulator.md`, по образцу `docs/07_chain.md`): формат
  сценария, HTTP-контракт стенда, формат трейса, формат отчёта проверки.
  **Черновик готов** (2026-07-15) — см. `docs/09_scenario_emulator.md`. В
  ходе черновика решены два вопроса, не покрытых списком «Подтверждённые
  решения» выше:
  - **Формат сценария** — YAML-frontmatter (`stand_url`, `max_turns`,
    `exact_checks`) + свободный текст тела (цель диалога, семантические
    критерии) — примиряет решение 1 (свободная форма) с решением 7 (точные
    ассерты без LLM нужно доставать машиночитаемо).
  - **Sandbox для Фазы 1** — контейнер (Docker) с сетью, ограниченной до
    адреса стенда.
  - Плюс критерий остановки цикла: `max_turns` в frontmatter (жёсткий
    детерминированный лимит, не «модель сама решает, что цель достигнута» —
    это смешало бы раннер с семантической проверкой Фазы 3).
  Часть содержимого документа (список типов `exact_checks`, layout `runs/`)
  помечена в самом документе как **предложение, ещё не согласованное** —
  подлежит подтверждению или правке до/во время Фазы 2.
- **Фаза 1** — фейковый эхо-стенд (HTTP, минимальный) + sandbox-окружение
  для эмулятора (сеть только к стенду) + cli-обвязка с внешней историей +
  запись трейса. Цель: сквозной прогон сценарий → диалог со стендом →
  трейс. **Код готов** (2026-07-15), см. `docs/09_scenario_emulator.md`,
  раздел «Реализация Фазы 1». Два уточнения решения 5 сделаны по ходу:
  cli-вызовы (генерация реплик) — на хосте, вне контейнера (аут `claude -p`
  не переносим в Linux-контейнер без доп. работы); сендбокс-канал до
  стенда — iptables egress-filter в relay-контейнере, а не Docker
  `--internal`-сеть (нужен для произвольного `stand_url`, не только
  фикстуры). **Оба пути (`--no-sandbox` и `--sandbox`) проверены живыми
  прогонами 2026-07-15** (реальный `claude -p` + фейковый эхо-стенд,
  `examples/scenarios/echo-smoke.md`, `run_status: ok`, 3 хода, трейс и
  финальный `get_state` совпадают с ожиданиями — модель даже сама
  заметила на третьем ходу, что говорит с эхо-стендом). Для `--sandbox`
  отдельно проверена сама изоляция (не только что прогон прошёл):
  `iptables`/`ip6tables` внутри relay-контейнера после lockdown — policy
  DROP + точечный ACCEPT только на стенд, попытка достучаться до
  постороннего хоста заблокирована.
- **Фаза 2** — точные проверки трейса (без LLM). **Готова и проверена**
  (2026-07-15) — `scenario_emulator/checks.py`, три типа `exact_checks` из
  `docs/09_scenario_emulator.md` (`message_contains`, `event_field`,
  `event_count`), встроена в `runner.py`/`report.json`. Живой прогон на
  `examples/scenarios/echo-smoke.md` подтвердил и проходной, и провальный
  случай (код возврата `run_scenario.py` — 0/1 соответственно).
- **Фаза 3** — семантическая проверка через `judge_client` (переиспользовать
  паттерн `_judge_transcript`). **Готова и проверена** (2026-07-15) —
  `scenario_emulator/semantic.py`, forced tool_use `submit_verdict`
  (`held`/`reason`), не вслепую — судье целиком показывается тело сценария
  как критерии. Не опциональна (в отличие от `exact_checks`) — тело
  сценария обязательно, судье всегда есть что оценить. Живой прогон
  подтвердил и `held: true` (эхо-смоук), и `held: false` (сценарий с
  намеренно противоречивыми критериями) с содержательным `reason` в обоих
  случаях; код возврата `run_scenario.py` учитывает `held` наравне с
  `exact_passed`.
- **Фаза 4** — CLI-обвязка, фикстуры/примеры, README/AGENTS.md. **Готова**
  (2026-07-15). CLI (`run_scenario.py`) и фикстура
  (`examples/scenarios/echo-smoke.md`) были готовы уже с Фазы 1 — довели до
  конца оставшееся:
  - Регрессия фикстур — **отдельный** новый skill
    `.claude/commands/scenario-regress.md`, НЕ секция внутри `judge-regress`
    (согласовано явно: `judge-regress` — чётко про `run_gates.py`+оракул,
    эмулятор сценариев архитектурно отдельный по решению 8, а каждый прогон
    фикстуры — реальные платные вызовы `claude -p`, дороже большинства
    проверок `judge-regress`). Оракул отдельным файлом не нужен — сценарий
    сам себе оракул через `exact_checks`+семантику, "правильный" результат
    фикстуры — код возврата `run_scenario.py` = 0. Скрипт skill'а прогнан
    живьём — фикстура проходит, `RUN_DIR`/report.json на месте.
