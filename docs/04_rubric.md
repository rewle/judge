# Гейт 04 — rubric (LLM-judge)

**Детерминизм:** нет — LLM-судья (Anthropic API, нативный SDK).
**Статус:** реализован (`gates/g04_rubric.py`), требует судью — api-ключ
или `JUDGE_BACKEND=cli` (см. README); без того и другого возвращает
`not_configured` (не блокирует цепочку, но явно виден в отчёте). Прогнан
на `golden-skill` через cli-бэкенд (2 прогона вместо 8): PASS, 0.80 ± 0.004
— логика сборки/усреднения/сигмы отработала штатно.
**Основа:** методология из `~/Projects/archive/skill_eval_dlq.ipynb`
(см. память `reference-skill-eval`).
**Когда гонять:** не на каждый push — по PR помечен `ready-for-review`, или
батчем раз в день по открытым PR (дорого, недетерминированно).

## Критерии (0.0–1.0 каждый)
Хранятся в `policies/rubric_criteria.yaml`, не в коде — правится без правки
гейта. Существующие 6 из ноутбука (conciseness, description_quality,
degrees_of_freedom, concrete_examples, consistent_terminology,
actionable_workflow) + новый для этого проекта:

- **groundedness** — содержит ли тело утверждения, которые нельзя
  проверить (придуманные факты, флаги, API), поданные уверенно, без пометки
  "не проверено"/"уточнить"; инструктирует ли скилл модель угадывать вместо
  проверки реального состояния (см. фикстуру `hallucination-prone-skill`)

## Отличия от ноутбука-первоисточника
- **Structured output вместо regex.** Ноутбук доставал JSON из свободного
  текста регэкспом (`re.search(r"\{.*\}", ...)`) — хрупко для CI-гейта.
  Здесь — forced tool-use с `input_schema`, ответ приходит уже
  провалидированным по схеме.
- **Свой клиент, не через прокси.** Ноутбук ходил через внутренний прокси
  VK (OpenAI-совместимый клиент). Здесь — нативный `anthropic` SDK,
  креды и (опционально) свой `base_url` только через env
  (`JUDGE_API_KEY`/`ANTHROPIC_API_KEY`, `JUDGE_BASE_URL`) — см. README.

## Прогон
8 параллельных запусков судьи (`ThreadPoolExecutor`, `judge.n_runs` в
`config.yaml`). На каждом прогоне считается один общий балл (среднее по
всем критериям), затем по 8 таким прогонам — mean и sigma (`overall_sigma`,
`statistics.stdev` — выборочное отклонение, не популяционное: n=8 это
выборка). Именно так это было откалибровано в ноутбуке (`sigma < 0.02` —
хорошо).

**Плюс `per_criterion[c].sigma`** (реализовано): усреднение по критериям
внутри прогона перед подсчётом `overall_sigma` может маскировать
нестабильность одного конкретного критерия — судья мечется 0.1↔0.9 по
`groundedness`, но стабилен по остальным пяти, а `overall_sigma` выйдет
обманчиво низкой. Поэтому sigma считается ещё и по каждому критерию
отдельно, на тот же порог `rubric_max_sigma` — не новый порог, та же
цифра применена к более чувствительной статистике.

Fail если `overall_mean < thresholds.rubric_min_score` ИЛИ
(`overall_sigma > thresholds.rubric_max_sigma` ИЛИ хотя бы один критерий
нестабилен по своей sigma) — нестабильность судьи, общая или по одному
критерию, тоже сигнал, не игнорировать.

## Reason в сообщении (реализовано)
Судья возвращает `reason` по каждому критерию в каждом из 8 прогонов —
раньше эта причина нигде не сохранялась, FAIL показывал голое число без
единого слова объяснения. Теперь для каждого критерия хранится
**представительная** reason (из прогона, чей score ближе всего к среднему
по этому критерию — не все 8 сразу, избыточно), а сообщение FAIL по
низкому баллу включает два худших критерия с их reason напрямую в
`message`, не только в `details.per_criterion`.

## История прогонов и дрифт (реализовано)
Каждый прогон дописывает строку в `history/rubric_runs.jsonl`
(`{skill, model, backend, timestamp, overall_mean, overall_sigma}`) — append-only,
без перезаписи. `skill` — это `frontmatter.name`, не путь и не имя папки
(переживает переезд фикстуры). `backend` (`api`/`cli`) — часть ключа
сравнения: сравнивать дрифт между api и cli некорректно, у cli другой путь
получения structured output.

При следующем прогоне той же связки skill+model+backend гейт находит
последнюю предыдущую запись и кладёт `drift_vs_previous` в `details`
всегда (просто число). Если `|drift| >= thresholds.rubric_drift_warn_abs`
(0.1) — в `message` добавляется пометка `[дрифт ... — проверить вручную]`.
**Это не отдельный fail-гейт** — дрифт не блокирует цепочку, только
подсвечивается: нет откалиброванного порога, за которым дрифт значим, а не
шум обычной нестохастичности judge (см. sigma выше). Проверено на реальных
последовательных прогонах через `JUDGE_BACKEND=cli`: два прогона подряд на
`golden-skill` (0.764 → 0.781, drift +0.017, ниже порога — пометка не
появилась, как и ожидалось).

## Известные ограничения / открытые вопросы
- **Модель-судья.** Дефолт `claude-sonnet-5` (переопределяется
  `JUDGE_MODEL`). Проверено на реальном прогоне только через cli-бэкенд
  (без платного ключа) — api-бэкенд с настоящим ключом не проверен.
- Порог `rubric_min_score=0.7` — взят по аналогии с эталоном
  `ingest-dlq-analysis` (rubric 0.87 на другой модели/промпте через прокси
  VK), нужно откалибровать заново на этой связке модель+промпт.
- Остаточная нестохастичность при формально «одной» модели — известное
  свойство LLM, поэтому и есть sigma-проверка, а не одноразовый вызов.
- **`rubric_drift_warn_abs=0.1`** — первое разумное значение, не
  откалибровано (как и остальные пороги 03-05).
- История хранится локально в git, не централизованно — если реестр
  скиллов реальный и прогоны идут из CI на разных раннерах/веток,
  `history/rubric_runs.jsonl` разъедется без внешнего хранилища
  (БД/S3/артефакты CI) — для локальной обкатки достаточно, для прод-CI нет.
