# Гейт 04 — rubric (LLM-judge)

**Детерминизм:** нет — LLM-судья (Anthropic API, нативный SDK).
**Статус:** реализован (`gates/g04_rubric.py`), требует ключ — без него
возвращает `not_configured` (не блокирует цепочку, но явно виден в отчёте).
**Основа:** методология из `~/Projects/archive/skill_eval_dlq.ipynb`
(см. память `reference-skill-eval`).
**Когда гонять:** не на каждый push — по PR помечен `ready-for-review`, или
батчем раз в день по открытым PR (дорого, недетерминированно).

## Критерии (0.0–1.0 каждый)
Хранятся в `policies/rubric_criteria.yaml`, не в коде — правится без правки
гейта. Существующие 6 из ноутбука (conciseness, description_quality,
degrees_of_freedom, concrete_examples, consistent_terminology,
actionable_workflow) + новый для этого проекта:

- **groundedness** — содержит ли тело утверждения, которые нельзя
  проверить (придуманные факты, флаги, API), поданные уверенно, без пометки
  "не проверено"/"уточнить"; инструктирует ли скилл модель угадывать вместо
  проверки реального состояния (см. фикстуру `hallucination-prone-skill`)

## Отличия от ноутбука-первоисточника
- **Structured output вместо regex.** Ноутбук доставал JSON из свободного
  текста регэкспом (`re.search(r"\{.*\}", ...)`) — хрупко для CI-гейта.
  Здесь — forced tool-use с `input_schema`, ответ приходит уже
  провалидированным по схеме.
- **Свой клиент, не через прокси.** Ноутбук ходил через внутренний прокси
  VK (OpenAI-совместимый клиент). Здесь — нативный `anthropic` SDK,
  креды и (опционально) свой `base_url` только через env
  (`JUDGE_API_KEY`/`ANTHROPIC_API_KEY`, `JUDGE_BASE_URL`) — см. README.

## Прогон
8 параллельных запусков судьи (`ThreadPoolExecutor`, `judge.n_runs` в
`config.yaml`). На каждом прогоне считается один общий балл (среднее по
всем критериям), затем по 8 таким прогонам — mean и sigma. Именно так это
было откалибровано в ноутбуке (`sigma < 0.02` — хорошо), поэтому сигма
считается на уровне прогона, а не отдельно по каждому критерию.

Fail если `overall_mean < thresholds.rubric_min_score` ИЛИ
`overall_sigma > thresholds.rubric_max_sigma` (нестабильность судьи —
тоже сигнал, не игнорировать).

## Известные ограничения / открытые вопросы
- **Модель-судья.** Дефолт `claude-sonnet-5` (переопределяется
  `JUDGE_MODEL`). Не проверено на реальных прогонах в этом окружении — нет
  доступного ключа на момент реализации.
- Порог `rubric_min_score=0.7` — взят по аналогии с эталоном
  `ingest-dlq-analysis` (rubric 0.87 на другой модели/промпте через прокси
  VK), нужно откалибровать заново на этой связке модель+промпт.
- Остаточная нестохастичность при формально «одной» модели — известное
  свойство LLM, поэтому и есть sigma-проверка, а не одноразовый вызов.
- История прогонов рубрики по каждому скиллу (дрифт судьи со временем при
  обновлении модели) — пока не хранится, гейт stateless.
