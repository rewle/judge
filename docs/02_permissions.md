# Гейт 02 — permissions

**Детерминизм:** полный, без внешних вызовов.
**Реализация:** `gates/g02_permissions.py`
**Политика:** `policies/tools_allowlist.yaml` (глобальная) +
`policies/systems/*.yaml` (по файлу на систему, мёрджатся при загрузке)
**Когда гонять:** каждый push (после 01).

## Модель
Deny-by-default. Три списка в политике:
- `allowed_by_default` — не требуют обоснования (Read, Grep, Glob)
- `requires_justification` — допустимы только если в frontmatter скилла
  есть непустое поле `justification`, объясняющее, зачем инструмент нужен
  именно этой задаче
- `always_flag_for_review` — паттерны (fnmatch), которые ВСЕГДА уходят на
  ручное ревью гейта 06 (`redteam`), даже если formально с justification.
  Например `Bash(*)` (неограниченный shell) или `mcp__*__delete*`.
  Существует ТОЛЬКО в глобальной политике — файл системы его не задаёт.

## Политика системы (решение 1C, для владельца системы)
Владелец корпоративной системы держит СВОЙ файл
`policies/systems/<system>.yaml` и не может задеть чужой:

```yaml
system: dwh                # обязан совпадать с именем файла
namespace: mcp__dwh__*     # mcp__<system>__*, границы владения
allowed_by_default:
  - mcp__dwh__list_tables
requires_justification:
  - mcp__dwh__query
  - mcp__dwh__export_*     # записи могут быть fnmatch-паттернами (1A)
```

При загрузке файл проходит «гейт для политик»: записи не могут выйти за
`namespace`; `always_flag_for_review` в файле системы запрещён
(предохранитель глобальный — согласуется с админом реестра); namespace'ы
двух файлов не могут пересекаться (одна система — один владелец); форма
проверяется так же, как у глобального файла. Живой пример —
`policies/systems/dwh.yaml`. Быстрая проверка своей правки:
`python3 run_gates.py --check-policy` (без прогона скиллов). Негативные
фикстуры всех веток валидации — `examples/policies/*` с оракулом
`expected_policy_results.yaml` (прогоняются `--check-policy <корень>`,
в живой `policies/` их не класть).

Паттерн шире задуманного (`mcp__dwh__*` вместо `mcp__dwh__get_*`) НЕ
открывает send/delete: предохранитель на рантайме срабатывает раньше
allow-списков, поэтому широкий паттерн не дыра, а лишь несбывшееся
ожидание владельца — автор скилла всё равно увидит «ручное ревью».
Формальную сверку паттернов политики с паттернами предохранителя гейт
сознательно не делает: у глобов почти любая пара формально пересекается
(строка `mcp__dwh__get___delete_x` подходит и `mcp__dwh__get_*`, и
`mcp__*__delete*`), и статическая проверка давала бы сплошные ложные
срабатывания; мёртвые записи ловятся только для литеральных имён.

## Вход
Frontmatter скилла, поле `tools:` (список или CSV-строка) +
`justification:` — строка (blanket на весь скилл) ИЛИ карта
`{инструмент_или_паттерн: причина}` (решение 3B): новичок пишет строку,
эксперт различает причины per-tool. Для инструмента из
`requires_justification` при карте ищется сначала точный ключ, потом
паттерн-ключ; дыра в карте — FAIL с именем инструмента (фикстуры:
`justification-map-skill` / `justification-map-gap-skill`).

## Выход
FAIL, если: инструмент вне всех списков; инструмент из
`requires_justification` без заполненного `justification`; инструмент
подпадает под `always_flag_for_review` (даже с обоснованием — это
сознательный hard-stop, ручное решение не автоматизируется этим гейтом).

## Регистрация корпоративного MCP-инструмента (частый случай эксперта)
Инструменты незнакомого MCP-сервера не входят ни в один список политики,
поэтому падают на ветке "не в allowlist и не в requires_justification".
**Поле `justification` на них не действует, пока инструмент не внесён в
политику** — justification снимает барьер только для инструментов из
`requires_justification`. Правильное место регистрации — файл политики
своей системы `policies/systems/<system>.yaml` (см. выше; владелец системы
регистрирует сервер паттерном одной строкой). Глобальный
`tools_allowlist.yaml` — для встроенных инструментов и предохранителя.
Сообщение гейта называет конкретный инструмент и этот следующий шаг.

## Следствие для группы скиллов: почему `send*`/`delete*` не автоматизируется
Если хоть один скилл группы (или транзитивно достижимый через `uses:`)
использует инструмент под `always_flag_for_review` (`mcp__*__send*`,
`mcp__*__delete*`, `Bash(*)`), то и он, и всякий оркестратор над ним
**всегда** упираются в ручное ревью — обоснование этот флаг не снимает
(см. "Незакрытый процессный вопрос" ниже). Практический вывод для эксперта,
собирающего кросс-системную группу: группа, которая доставляет результат
(шлёт письмо, пишет в чужую систему), не пройдёт пайплайн на зелёный
автоматически by design. Это стоит закладывать в план заранее, а не узнавать
на гейте.

## Валидность самой политики (зона владельца системы)
Политику правят владельцы систем разной квалификации (см. `docs/personas.md`,
портрет 3), поэтому `_load_policy()` валидирует сам файл политики и при
проблеме гейт падает с сообщением, атрибутированным **политике, а не скиллу**
(«это проблема политики, не скилла», с номером строки). До этого битая
политика либо роняла раннер traceback'ом, либо молча парсилась «не так» — и
гейт советовал автору скилла добавить инструмент, который владелец только что
добавил.

Проверяется:
- YAML парсится; верхний уровень — маппинг.
- Все три ключа обязательны; сознательно пустой список пишется явно
  (`always_flag_for_review: []`), чтобы предохранитель нельзя было снять
  «незаметно удалив ключ». Неизвестные ключи — ошибка с подсказкой ближайшего
  известного (ловит опечатки типа `alowed_by_default`).
- Значения — списки строк: CSV-строка (`allowed_by_default: Read, Grep`) —
  ошибка (в отличие от `tools:` в frontmatter скилла, где CSV допустим);
  запись с пробелом — ошибка «похоже, две строки склеились из-за отступа».
- Противоречия: инструмент в обоих списках сразу; «мёртвая запись» —
  инструмент в `allowed_by_default`/`requires_justification`, подпадающий под
  паттерн `always_flag_for_review` (предохранитель срабатывает раньше обоих
  списков, поэтому такая запись только обещает то, чего гейт никогда не
  сделает — сам порядок «сначала предохранитель» при этом не меняется).

Для файлов систем (`policies/systems/*.yaml`) действует дополнительный
слой валидации — см. «Политика системы» выше: namespace-границы,
запрет на `always_flag_for_review`, непересечение namespace'ов. Внутри
глобального файла разделения ролей нет by design: его сопровождает админ
реестра, а не владельцы систем.

## Почему это не судья, а статика
Права декларируются явно в frontmatter — это не то, что требует понимания
смысла текста, поэтому гейт держим детерминированным и дешёвым, а не отдаём
LLM-судье.

## Переиспользование
`evaluate_tools(tools, justification, policy)` — движок политики, вынесен
из `check()` отдельной функцией. Гейт 07 (`docs/roadmap_chains.md`)
прогоняет через неё не только собственные `tools:` скилла, а объединение
по транзитивному замыканию `uses:` — так ловится "A с виду безобидный, но
через `uses: [B]` эффективно получает `Bash(*)` от B", что этот гейт
(видит только один скилл) увидеть не может в принципе.

## Открытый вопрос
Список `tools:` в frontmatter — это то, что скилл *просит* явно, а не
гарантия, что тело скилла не подговаривает модель попросить/использовать
что-то ещё в рантайме — ни через собственные инструменты (зона гейта 06,
redteam), ни через вызов скилла, не задекларированного в `uses:` (зона
гейта 07, обратная проверка).

## Незакрытый процессный вопрос: кто делает ручное ревью
`always_flag_for_review` — намеренный hard-stop: сообщение гейта явно
говорит "требует ручного review человеком вне этого пайплайна", но проект
**не определяет, кто именно** этот человек и **как** его запросить (нет
роли/канала/процесса эскалации ни в этом репозитории, ни в доках). Пока
это не решено на уровне конкретного реестра/организации, PR с
`Bash(*)`/`mcp__*__delete*`/и т.п. упирается в тупик без описанного
следующего шага — это известное ограничение, не спроектированное
"фичей отсутствия".
