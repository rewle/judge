# Эмулятор пользователя по декларативному сценарию

**Статус:** контракт Фазы 0 + реализация Фазы 1 (сквозной прогон
сценарий → диалог со стендом → трейс) — **готова и проверена живыми
прогонами 2026-07-15**, оба пути: `--no-sandbox` (прямой HTTP) и
`--sandbox` (iptables-relay, реальный Docker daemon) —
`examples/scenarios/echo-smoke.md`, реальный cli-бэкенд, 3 хода, трейс и
`get_state` корректны в обоих случаях (модель даже сама заметила на
третьем ходу, что говорит с эхо-стендом). Для `--sandbox` отдельно
проверена сама изоляция (не только что прогон прошёл): `iptables -L`/
`ip6tables -L` внутри relay-контейнера после lockdown показывают
`policy DROP` + точечный `ACCEPT` только на IP:port стенда, попытка
достучаться до постороннего хоста (`1.1.1.1`) заблокирована. Фазы 2
(точные проверки трейса, `scenario_emulator/checks.py`) и 3 (семантическая
оценка, `scenario_emulator/semantic.py`) тоже готовы и проверены живыми
прогонами (проходной и провальный случаи для обеих). Фаза 4 (CLI, фикстуры,
регрессия, README/AGENTS.md) тоже готова — регрессия отдельным skill'ом
`/scenario-regress`, не внутри `/judge-regress`. Не гейт в
цепочке `config.yaml` — отдельный инструмент рядом с `run_gates.py`
(подтверждённое решение 8, см. `docs/TODO_scenario_emulator.md`). Формат
файла — по конвенции остальных гейтов (`docs/<NN>_<name>.md`), но раздела
«Пороги» в привычном смысле нет: это не judge-скоринг с калибруемым порогом,
а раннер + двухчастная проверка.

Предыстория, происхождение идеи и список из 8 подтверждённых решений —
`docs/TODO_scenario_emulator.md`. Этот документ их не повторяет, а
конкретизирует до формата, достаточного для реализации Фазы 1.

## Область действия

Эмулятор ведёт диалог с внешней системой под тестом («стенд») от лица
подставного пользователя, следуя цели из сценария, и проверяет результат
двумя способами: точно (assert по записанному трейсу, без LLM) и
семантически (через `judge_client`, по аналогии с `_judge_transcript` из
`gates/g06_redteam.py`).

Независимость от графа `uses:`/chains/groups (гейт 07) — временное
допущение, см. TODO, не пересматривается в этом документе.

## Формат сценария

YAML-frontmatter + свободный текст тела — по аналогии с `SKILL.md` и
`group.yaml` (frontmatter — то, что должно парситься детерминированно;
текст — то, что читает человек и judge). Выбрано в обсуждении 2026-07-15
как способ примирить решение 1 («сценарий свободной формы») с решением 7
(«точная сверка трейса без LLM» требует машиночитаемых ассертов).

```yaml
---
stand_url: http://localhost:8765
max_turns: 6
exact_checks:
  - type: message_contains
    role: assistant
    pattern: "счёт закрыт"
  - type: event_field
    event: stand_call
    handle: get_state
    path: session.status
    equals: closed
  - type: event_count
    event: stand_call
    handle: send_message
    max: 6
---
Цель диалога: пользователь просит закрыть счёт 123 и уточняет комиссию.

Семантически проверить: ассистент вежливо объяснил последствия закрытия
счёта, не пообещал невозможного (например, отмену уже прошедших платежей).
```

- `stand_url` — обязателен, адрес стенда для этого прогона.
- `max_turns` — обязателен (решение от 2026-07-15): жёсткий детерминированный
  предел числа обменов сообщениями «эмулятор ↔ стенд» за прогон. Критерий
  остановки цикла Фазы 1 — не «цель диалога достигнута» (это смешало бы
  раннер с семантической проверкой Фазы 3), а исчерпание лимита либо более
  ранняя естественная точка (например стенд сам сигнализирует конец сессии
  через `get_state`, если это предусмотрено его контрактом). При достижении
  `max_turns` без естественного завершения раннер просто останавливается и
  передаёт трейс на проверку как есть — упор в лимит сам по себе не FAIL,
  это решают exact_checks/семантика по факту содержимого трейса.
- Тело (всё после второго `---`) целиком идёт в семантический слой (Фаза 3,
  judge) как описание цели диалога и критериев — свободный текст, не
  парсится.
- `exact_checks` — список ассертов Фазы 2, каждый проверяется без LLM
  против записанного трейса прогона (не напрямую против стенда — трейс уже
  содержит всё, что стенд возвращал по ходу диалога, см. «Формат трейса»
  ниже).

### Типы `exact_checks` (v0, список закрытый и расширяемый по потребности)

| type | поля | проверяет |
|---|---|---|
| `message_contains` | `role`, `pattern` (regex) | хотя бы одно сообщение данной роли в трейсе матчит паттерн |
| `event_field` | `event`, `handle`, `path` (dot-path), `equals`/`contains` | поле внутри `body` последнего (или указанного `index`) события с данными `event`+`handle` |
| `event_count` | `event`, `handle`, `min`/`max` | число событий данного вида в трейсе в границах |

`event_field` — обобщение примера с `tool_called` из обсуждения: вместо
жёсткого предположения, что стенд возвращает именно «вызовы тулов», проверка
адресует произвольный JSON, который стенд вернул на `get_state` (или любой
другой хендл) — путь и значение сравниваются как есть, схема ответа стенда
не стандартизируется (см. «HTTP-контракт стенда»).

**Это предложение, не зафиксированное в решениях 2026-07-15** — состав
списка и имена типов не обсуждались явно, только сам принцип
frontmatter+текст. Если состав не подходит — скорректировать перед Фазой 2
(реализация exact-checker), это не блокирует Фазу 0/1.

## HTTP-контракт стенда

Минимальный набор хендлов (решение 2): открыть сессию, отправить сообщение,
получить состояние. Конкретные пути/методы ниже — тоже предложение
Фазы 0, не отдельно согласовывались:

```
POST /session
  → 200 {"session_id": "<str>"}

POST /session/{session_id}/message
  body: {"content": "<str>"}
  → 200 {"content": "<str>"}          # ответ стенда пользователю

GET /session/{session_id}/state
  → 200 <произвольный JSON>            # схема — на усмотрение стенда,
                                        # exact_checks (event_field) адресуют
                                        # её через dot-path, не через
                                        # предопределённый контракт полей
```

Закрытие сессии не входит в минимальный набор (решение 2 перечисляет ровно
три хендла) — если стенду нужна очистка, это его забота (например TTL),
эмулятор не обязан вызывать `DELETE`.

Любой не-200 ответ стенда — событие в трейсе с телом ошибки, весь прогон
сценария завершается неуспехом на уровне раннера (до judge дело не доходит)
с отдельным статусом, не смешиваемым с `FAIL` exact/semantic слоёв
(см. «Формат отчёта»).

## Ядро эмулятора (уточнение решения 4)

Каждый вызов к cli-бэкенду через `judge_client.ClaudeCLIClient` — один
`user`-message без истории (ограничение `_CLIMessages.create`,
`judge_client.py:100-103`). Решение 4 говорит «историю ведём сами, на
каждый вызов передаём накопленный контекст целиком» — технически это
означает: на каждом шаге эмулятор формирует ОДНО user-сообщение, в которое
целиком сериализован накопленный диалог (роли, реплики, ответы стенда) плюс
текст сценария в system-промпте, а не использует `messages` как массив из
нескольких сообщений. Многосообщенческая история `messages` бэкендом не
поддерживается и не нужна — это не смягчение решения 4, а способ его
реализовать в рамках существующего интерфейса `judge_client`, без изменений
в `judge_client.py`.

## Формат трейса

Append-only JSONL (решение 6) — по конвенции `history/rubric_runs.jsonl`
(единственный уже существующий precedent append-only лога в проекте).
Каждая строка — одно событие:

```json
{"ts": "2026-07-15T12:00:00.000Z", "type": "user_message", "content": "..."}
{"ts": "2026-07-15T12:00:01.000Z", "type": "stand_call", "handle": "send_message", "request": {...}, "response": {...}, "status": 200}
{"ts": "2026-07-15T12:00:02.000Z", "type": "assistant_message", "content": "..."}
{"ts": "2026-07-15T12:00:03.000Z", "type": "stand_call", "handle": "get_state", "request": {}, "response": {...}, "status": 200}
```

`event_field`/`event_count` в exact_checks матчят по `type` (= `event` в
таблице выше, переименовано здесь для отличия от Python-типа) и `handle`.

Файл трейса не в `history/` (та папка — агрегатные логи по всем прогонам
гейтов, не по одному прогону сценария) — предлагается отдельная директория
`runs/` в корне репозитория, один подкаталог на прогон:
`runs/<scenario-name>_<timestamp>/trace.jsonl` +
`runs/<scenario-name>_<timestamp>/report.json` (см. ниже). Эфемерные
артефакты прогона, не фикстуры — не должны коммититься; добавить в
`.gitignore`.

## Формат отчёта проверки

```json
{
  "scenario": "close-account",
  "stand_url": "http://localhost:8765",
  "run_status": "ok",              // ok | stand_error | timeout
  "exact_checks": [
    {"type": "message_contains", "passed": true, "detail": "..."},
    {"type": "event_field", "passed": false, "detail": "session.status = open, ожидалось closed"}
  ],
  "exact_passed": false,
  "semantic": {"held": true, "reason": "..."},   // Фаза 3, судья
  "trace_path": "runs/close-account_20260715T120000Z/trace.jsonl"
}
```

Точный и семантический слои — раздельные поля, не сворачиваются в один
булев вердикт: пользователь инструмента должен видеть, какой из двух
провалился, а не только общий FAIL (тот же принцип, что у гейта 07 — I/O и
токен-бюджет не смешиваются в одну строку).

## Sandbox стенда для Фазы 1 (уточнение решения 5)

Решено в обсуждении 2026-07-15, в два захода — оба сузили решение 5 до
конкретного механизма:

**1. cli-вызовы (генерация реплик) — вне контейнера.** `claude -p`
аутентифицируется локальной OAuth/подпиской, не переносимым API-ключом —
перенести этот аут в изолированный Linux-контейнер для Фазы 1 не делаем.
Поэтому сендбоксируется НЕ весь раннер, а только канал раннер↔стенд:
- Генерация реплик (`judge.messages.create`, cli-бэкенд) — на хосте, без
  сетевых ограничений.
- HTTP-обмен со стендом (`POST /session`, `POST /session/{id}/message`,
  `GET /session/{id}/state`) — через relay-контейнер (см. ниже).

Это осознанное **сужение** решения 5 для Фазы 1, не полная его реализация:
если бы сам цикл, потребляющий ответы стенда, был скомпрометирован через
контент стенда (класс риска, похожий на гейт 06 redteam), защиты от этого
в Фазе 1 нет — изолирован только транспорт до стенда, не логика раннера
вокруг него. Полная песочница для LLM-цикла целиком — не сделана,
последующий шаг, если понадобится.

**2. Механизм канала — iptables egress-filter внутри relay-контейнера**
(`scenario_emulator/docker_sandbox.py`, `scenario_emulator/sandbox_relay.py`,
`docker/relay.Dockerfile`, `docker/relay-entrypoint.sh`), а не Docker
`--internal`-сеть. Причина выбора: `--internal`-сеть работала бы только
если стенд тоже контейнеризован и подключён к той же сети — то есть только
для собственного фейкового эхо-стенда, а не для «любого внешнего сервиса»
(решение 2). iptables-вариант работает для произвольного `stand_url`:
- Хост резолвит хост стенда в IP ДО старта контейнера (либо использует
  специальное имя `host.docker.internal`, если стенд на `localhost` —
  единственное имя, которое контейнеру разрешено резолвить самому при
  старте, см. `_resolve_stand_host`).
- Контейнер получает IP+порт стенда через env, при старте (entrypoint)
  выставляет `iptables`/`ip6tables`: DROP всё исходящее по умолчанию,
  точечный ACCEPT только на этот IP:port (+ loopback, + established/related
  для ответного трафика), затем запускает relay-процесс
  (`scenario_emulator/sandbox_relay.py`) поверх stdin/stdout — один
  контейнер на весь прогон сценария, не один `docker run` на ход диалога.
- Раннер общается с relay-контейнером через `SandboxedStandClient` —
  тот же интерфейс `.call()`, что у `DirectStandClient` (без сендбокса,
  `--no-sandbox`), так что `scenario_emulator/runner.py` не знает, каким
  путём ушёл запрос.

**Проверено живым прогоном 2026-07-15** (Docker Desktop поднят владельцем в
ходе той же сессии): `NET_ADMIN`/`NET_RAW` в контейнере работают,
`host.docker.internal` резолвится и достижим, `iptables -L -n -v`/
`ip6tables -L -n -v` внутри контейнера после lockdown-скрипта показывают
`Chain OUTPUT (policy DROP)` с единственным точечным ACCEPT на
`<IP стенда>:8765` (плюс loopback и ESTABLISHED/RELATED для ответов) —
IPv4 и IPv6 отдельно. Попытка изнутри контейнера достучаться до
постороннего хоста (`wget http://1.1.1.1`) заблокирована, как и
предполагалось. Полный прогон `run_scenario.py` (без `--no-sandbox`) через
этот же relay — `run_status: ok`, контейнер сам убирается за собой
(`--rm`), осиротевших контейнеров не остаётся.

## Реализация Фазы 1 (2026-07-15)

```
scenario_emulator/
  echo_stand.py      — фейковый эхо-стенд (решение 3), stdlib http.server
  scenario.py         — парсинг файла сценария (frontmatter + тело)
  trace.py            — TraceWriter (JSONL append) + read_trace
  stand_client.py      — DirectStandClient (.call(), без сендбокса)
  sandbox_relay.py     — процесс внутри relay-контейнера
  docker_sandbox.py    — SandboxedStandClient, хостовая обвязка docker run
  runner.py            — цикл диалога (cli-бэкенд judge_client, max_turns)
docker/
  relay.Dockerfile        — образ relay-контейнера
  relay-entrypoint.sh      — резолв IP стенда + iptables lockdown
run_scenario.py         — CLI (--scenario, --out-dir, --no-sandbox)
examples/scenarios/echo-smoke.md — фикстура для смоук-теста Фазы 1
```

Прогнать смоук-тест:
```bash
python3 -m scenario_emulator.echo_stand &            # фейковый стенд на :8765
python3 run_scenario.py --scenario examples/scenarios/echo-smoke.md --no-sandbox   # или без флага — через сендбокс
```
Оба режима — **проверены живым прогоном 2026-07-15** (реальный `claude -p`,
3 хода, `run_status: ok`, трейс и финальный `get_state` совпадают с
ожиданиями). Для `--sandbox` (по умолчанию, без флага) дополнительно
проверена сама сеть-изоляция, не только то, что прогон прошёл — см.
«Sandbox стенда для Фазы 1».

Ошибка ДО начала прогона (judge не сконфигурирован — нет `claude` в PATH
для cli-бэкенда) не попадает в `report.json` вообще: `run_scenario.py`
печатает причину в stderr и завершается с кодом 1 без файла отчёта.

## Реализация Фазы 2 (2026-07-15)

`scenario_emulator/checks.py` — `evaluate_checks(exact_checks, events)`
реализует все три типа из таблицы выше (`message_contains`, `event_field`,
`event_count`) буквально как описаны, без изменений состава. Вызывается из
`scenario_emulator/runner.py` (`_finish_report`) после завершения диалога —
**даже если `run_status != "ok"`**: частичный трейс (диалог оборвался
раньше срока) всё равно есть смысл сверить, `event_count` в этом случае
честно покажет недостачу событий. Если сценарий не задаёт `exact_checks` —
`exact_checks`/`exact_passed` в отчёте остаются `null` (нечего проверять,
как и раньше в Фазе 1). Ошибка формы самого `exact_checks` (опечатка в
`type`, отсутствует обязательное поле) — `CheckSpecError`, всплывает как
ошибка запуска (`RuntimeError`, до записи `report.json`), отдельно от
`passed: false` (провалившаяся, но корректно заданная проверка).

`run_scenario.py`: код возврата — `0` только если `run_status == "ok"` И
`exact_passed` не `False` — CI-подобное использование смотрит на код
возврата, не обязано парсить JSON.

Проверено живым прогоном 2026-07-15 (`--no-sandbox`, реальный `claude -p`):
`examples/scenarios/echo-smoke.md` — оба check'а (`message_contains`,
`event_count`) проходят, `exact_passed: true`, код возврата 0. Отдельно
проверен провальный путь на временном сценарии (`event_field` на
несуществующий путь ответа стенда, `event_count` с заведомо недостижимым
`min`) — `exact_passed: false`, диагностика в `detail` по каждому check'у
понятна без чтения кода, код возврата 1.

## Реализация Фазы 3 (2026-07-15)

`scenario_emulator/semantic.py` — `evaluate_semantic(client, model,
goal_text, events)`: тот же паттерн forced tool_use (`submit_verdict`,
поля `held`/`reason`), что `_judge_transcript` в `gates/g06_redteam.py`, но
**не вслепую** — в отличие от гейта 06 (судья не знает, что тестируется
конкретный скилл, это часть теста на инъекции), здесь судье намеренно
показывают весь текст тела сценария (`goal_text`) целиком как критерии —
оценка по определению делается относительно заявленного, а не «не поддался
ли инъекции». Принимает уже сконфигурированные `client`/`model` от раннера
(не создаёт свой get_client() — тот уже вызван для генерации реплик
диалога, см. «Ядро эмулятора»).

В отличие от `exact_checks` (опциональны — пропускаются, если сценарий их
не задал), semantic **не опционален**: тело сценария обязательно
(`Scenario.load` гарантирует непустой `goal_text`), значит судье всегда
есть что оценить — даже пустой/оборвавшийся диалог получает вердикт по
факту («диалог не начался» — тоже законное основание для `held: false`).

`run_scenario.py`: код возврата — `0`, только если `run_status == "ok"` И
`exact_passed` не `False` И (`semantic` пуст или `held` не `False`).

Проверено живым прогоном 2026-07-15 (`--no-sandbox`, реальный `claude -p`,
модель-судья — та же, что играет пользователя, в отдельном вызове):
`examples/scenarios/echo-smoke.md` — `held: true` с содержательным `reason`

вердикт, код возврата 1.
