# Портреты пользователей judge

Справочник четырёх ролей, работающих с сервисом `judge` (CI/CD-гейты для
реестра Claude Code скиллов). Документ нужен, чтобы в будущих сессиях можно
было «посмотреть на сервис глазами» конкретной роли при ревью изменений:
проверить, не ломает ли правка привычный путь новичка, композицию эксперта,
работу владельца системы или надзор администратора. Все четыре портрета —
выводы проведённых UX-прогонов (даты в пометках разделов).

Каждый портрет описан по единой структуре: кто это и цель, что трогает в
сервисе, что нужно от judge, известная фрикция (со ссылкой на гейт/файл) и
критерий успеха.

---

## Портрет 1. Новичок (нуб)

**Кто это и цель.** Пишет свой первый `SKILL.md`. Доки читает редко или не
читает вовсе, склонен сначала просто закоммитить и посмотреть, что
произойдёт, а разбираться — потом. Цель: получить рабочий скилл, принятый
реестром, с минимумом теории.

**Что трогает в сервисе.** `new_skill.py` (генератор скелета скилла),
`run_gates.py --path ./skill`. Из гейтов реально доходит до 01 (статика) и
02 (права) — дальше без ключа судьи обычно не идёт (гейты 04/05/06
возвращают `not_configured`).

**Что нужно от judge.** Понятные сообщения об ошибках; минимум концепций;
один очевидный путь без развилок; быстрый цикл «правка → проверка»; явный
сигнал «готово», чтобы не гадать, закончил ли он.

**Известная фрикция** (в основном уже устранена, статусы отмечены):
- (а) ГОТОВО — детектор плейсхолдеров гейта 01 раньше ловил любое слово
  `TODO`/`FIXME`/`TBD` и потому блокировал скилл, который сам ПРО извлечение
  TODO-комментариев (скилл не мог себя описать). Теперь детектор ловит
  только форму заметки-«доделать», а не упоминание термина.
- (б) ГОТОВО — сообщения гейта 01 теперь указывают номер строки и поле и
  перечисляют все совпадения. Раньше они не говорили ГДЕ проблема, и новичок
  чинил вслепую.
- (в) ГОТОВО — `run_gates.py` теперь печатает строку-вердикт при успехе и
  поясняет, что `SKIP` — это не провал, а пропуск (например, judge не
  сконфигурирован).
- (г) ГОТОВО — в README добавлено, что гейты не запускаются сами по
  `git commit`: в репозитории нет хука/CI, это библиотека под чужой CI.

  Итог правок: путь новичка сократился примерно с 6 итераций до 2–3.

**Критерий успеха.** За 2–3 прогона доходит либо до зелёного вердикта, либо
до понятного отказа — без ложных срабатываний и без необходимости читать
документацию.

## Портрет 2. Эксперт (профи)

**Кто это и цель.** Собирает ГРУППУ скиллов, объединяющую несколько
корпоративных систем с РАЗНЫМИ политиками доступа (пример: CRM — только
чтение; хранилище данных — доступ по обоснованию; почта — отправка от чужого
имени). Композиция выражается через конвенции `uses:` / `provides:` во
frontmatter. Цель: поставить группу как единицу и получить судейские
проверки её композиции.

**Что трогает в сервисе.** `policies/tools_allowlist.yaml` (глобальный
allowlist прав); конвенции frontmatter `uses:` / `provides:` /
`justification:`; гейт 07 (граф `uses:`, реестровая проверка); `config.yaml`
(пороги, `registry_dir`/`skills_dir`); `run_gates.py --check-chains
--registry`.

**Что нужно от judge.** Регистрировать корпоративные системы с политикой на
систему; выражать группу как единицу поставки; обосновывать права
per-tool/per-system; получать судейские проверки композиции
(I/O-совместимость, токен-бюджет цепочки, многошаговый prompt injection).

**Известная фрикция** (решения владельца 2026-07-13 реализованы — см.
`docs/design_multisystem_groups.md`; статусы отмечены):
1. ГОТОВО (1C) — появилась политика-на-систему: владелец регистрирует свой
   MCP-сервер в `policies/systems/<system>.yaml` (паттерном одной строкой),
   эксперту больше не нужно тащить каждый инструмент в глобальный файл.
2. ГОТОВО — сообщение про неизвестный инструмент называет инструмент и
   следующий шаг (файл своей системы или глобальную политику).
3. ЧАСТИЧНО, остальное by design — `always_flag_for_review` по-прежнему не
   обходится обоснованием (это предохранитель, не баг), но «узнать заранее
   нельзя» починено: проверяемый `group.yaml` (2B) заранее печатает «группа
   гарантированно потребует ручного ревью» с перечнем инструментов.
4. НЕ починено, отклонено владельцем — один заблокированный скилл гасит
   гейт 07 для всей группы; флаг «прогнать 07 при частичном фейле»
   отклонён сознательно (07 на заведомо битом графе).
5. НЕ починено (блокер ключа) — судейские слои гейта 07 без API-ключа
   «тёмные», см. открытый вопрос в портрете админа.
6. ГОТОВО (4A) — `--skills-dir ./my-group`: полный цикл по своей папке без
   правки конфига, гейт 03 ищет дубли внутри группы, гейт 07 берёт её как
   реестр (вместе с манифестом).
7. ГОТОВО (3B) — `justification` может быть картой {инструмент: причина};
   строка осталась blanket'ом для новичка.

**Что работает хорошо** (не ломать при правках):
- Транзитивная эскалация прав: гейт 07 ловит, что оркестратор, с виду
  запрашивающий только `Read`, через `uses:` стягивает почту + хранилище +
  CRM, и честно называет каждый инструмент и его происхождение.
- `always_flag` на `send`/`delete` как предохранитель.
- Лёгкие, читаемые конвенции `uses:` / `provides:`.

**Критерий успеха.** Может зарегистрировать системы и их политики без правки
кода гейтов; видит эскалации прав, циклы и несовместимости композиции;
заранее понимает, что именно потребует ручного ревью.

## Портрет 3. Владелец системы

> **UX-прогон проведён (2026-07-13).** Сценарии: внести инструменты своего
> MCP-сервера в политику и убедиться, что правка сработала; сломать YAML
> (сбитый отступ, таб); опечататься в ключе; написать строку вместо списка;
> внести свой delete-инструмент в allowed; удалить ключ предохранителя.
> Найденное и починенное — ниже; крупное остаётся в design-note
> `docs/design_multisystem_groups.md`.

**Кто это и цель.** Отвечает за одну корпоративную систему (например CRM):
поддерживает её MCP-сервер и **политику доступа к её инструментам** в
актуальном состоянии. Квалификация РАЗНАЯ от владельца к владельцу — но мы
можем ставить им ограничения и правила, поэтому низкая квалификация части
владельцев — меньшая из проблем (её гасят guardrail'ы и валидация с нашей
стороны, а не доверие). Цель: держать политику своей системы актуальной, не
задев чужие и не открыв дыру.

**Что трогает в сервисе.** Файл политики СВОЕЙ системы —
`policies/systems/<system>.yaml` (решение 1C: namespace, allow-списки,
паттерны; глобальный `tools_allowlist.yaml` и предохранитель — зона админа
реестра). Проверяет правку через `run_gates.py --check-policy`. Не трогает
код гейтов, не пишет скиллы.

**Что нужно от judge.** Простой ограниченный формат политики с понятными
ошибками (по сути тот же уровень заботы, что путь новичка, только про
политику, а не про скилл); изоляцию (правка политики CRM не ломает почту);
валидацию со стороны judge, которая не даёт владельцу с низкой квалификацией
сконфигурировать опасно (например обойти `always_flag_for_review` или
раздать права шире своего namespace).

**Известная фрикция** (по UX-прогону; статусы отмечены):
- (а) ГОТОВО — раньше саму политику никто не валидировал, и любая ошибка
  владельца была либо traceback'ом (таб/битый YAML ронял раннер), либо молчала
  и путала: сбитый отступ склеивал две записи в одну; опечатка в ключе
  (`alowed_by_default`) молча опустошала allowlist; строка вместо списка
  превращалась в набор символов. Во всех «молчащих» случаях гейт 02 винил
  СКИЛЛ и советовал «добавь инструмент в политику» — тот самый, который
  владелец только что добавил (замкнутый круг). Теперь `_load_policy`
  валидирует файл (см. `docs/02_permissions.md`, «Валидность самой
  политики»): ошибка атрибутирована политике, с номером строки и подсказкой
  (включая ближайший ключ при опечатке).
- (б) ГОТОВО — «мёртвые записи»: владелец вносил `mcp__crm__delete_deal` в
  `allowed_by_default`, предохранитель `mcp__*__delete*` на рантайме всё
  равно побеждал (это правильно и сохранено), но противоречие никто не
  называл — автор скилла упирался в ручное ревью, хотя политика «разрешала».
  Теперь противоречие — ошибка политики с явным текстом.
- (в) ГОТОВО — снятие предохранителя удалением ключа
  `always_flag_for_review` проходило молча (скилл с delete шёл на зелёный).
  Теперь все три ключа обязательны; сознательно пустой список — только явный
  `[]`.
- (г) ГОТОВО (решение 1C) — изоляция и namespace-границы: владелец правит
  только свой `policies/systems/<system>.yaml`; «гейт для политик» не даёт
  выйти за namespace, задать `always_flag_for_review` в файле системы
  (предохранитель глобальный, правит его только админ) или пересечься
  namespace'ом с чужой системой. См. `docs/02_permissions.md`.
- (д) ГОТОВО — `python3 run_gates.py --check-policy`: проверка своей правки
  политики (и конфига) без прогона скиллов, без скиллов-пустышек и стены
  FAIL'ов негативных фикстур.
- (е) ГОТОВО (1A внутри 1C) — в файле системы записи могут быть
  fnmatch-паттернами: сервер регистрируется строкой `mcp__dwh__export_*`,
  а не поимённо. Предохранитель на рантайме побеждает любой паттерн.

**Критерий успеха.** Владелец любой квалификации держит политику своей
системы актуальной за понятное число шагов, с внятными ошибками, и не может
случайно навредить чужим системам или снять предохранители реестра.

## Портрет 4. Админ реестра

> **UX-прогон проведён (2026-07-13).** Сценарии: сменить порог и увидеть
> эффект; опечататься в ключе порога; удалить порог; сломать YAML конфига;
> опечататься в module гейта; довериться секции `policies:`; разобрать
> инцидент по `history/` (дрифт, кэши).

**Кто это и цель.** Администратор самого пайплайна `judge`, **средний
уровень экспертизы в LLM**. Контролирует работу реестра оверол (в отличие от
владельца системы, отвечающего за одну систему): настраивает пороги и
глобальные политики, читает отчёты, разбирает инциденты, смотрит историю в
`history/`. Цель: держать пайплайн предсказуемым и объяснимым.

**Что трогает в сервисе.** `config.yaml` (пороги `thresholds`, порядок
гейтов); `policies/` (allowlist, `rubric_criteria`, `redteam_attacks`);
`history/` (append-only логи дрифта рубрики, кэши дублей и токен-бюджета);
`docs/` (контракты гейтов).

**Что нужно от judge.** Калибровать пороги; понимать отчёты; расследовать,
почему гейт сработал или не сработал.

**Известная фрикция** (по UX-прогону; статусы отмечены):
- (а) ГОТОВО — `config.yaml` никто не валидировал, ошибки админа вели себя
  как ошибки владельца политики до фикса: битый YAML / удалённый порог /
  опечатка в `module:` — голый traceback; опечатка в имени порога — МОЛЧА
  игнорировалась, гейт жил на зашитом в код дефолте, и админ не узнавал, что
  его порог не применился (`.get(key, default)` в гейтах). Теперь
  `load_config()` в `run_gates.py` валидирует конфиг: неизвестные ключи с
  подсказкой ближайшего, все пороги обязательны, module каждого гейта
  проверяется на импортируемость, ошибки с номером строки. Код выхода
  различает «скилл плохой» (1) и «пайплайн неисправен» (2) — в CI видно, чья
  зона.
- (б) ГОТОВО — секция `policies:` в конфиге была мёртвой ручкой: пути в ней
  не читал ни один гейт (захардкожены), админ, указавший другой файл, молча
  не менял ничего. Секция убрана, валидатор объясняет это старым конфигам.
- (в) ГОТОВО — контракт гейта 07 жил только внутри `docs/roadmap_chains.md`
  (дизайн-историей), нарушая правило AGENTS.md «у каждого гейта — контракт в
  docs/<NN>_<name>.md». Добавлен `docs/07_chain.md`.
- (г) Работает хорошо (не ломать): смена порога видна сразу (сообщения гейтов
  печатают и факт, и порог); кэши judge-вердиктов инвалидируются по хэшу
  содержимого и модели, а порог применяется ПОСЛЕ кэша — смена порога не
  требует чистки кэшей; дрифт рубрики виден прямо в сообщении гейта 04
  (non-blocking), история — `history/rubric_runs.jsonl`.

**Открытый вопрос** (не фрикция UX, а известный блокер). Пороги НЕ
откалиброваны без живого API-ключа (см. `AGENTS.md`): судье-зависимая зона
(04/05/06, judge-слои 03/07) для локального админа «тёмная» — та же пятая
фрикция эксперта. Калибровка — отдельный шаг (прогон на всех фикстурах через
`JUDGE_BACKEND=api`, сверка с `examples/expected_results.yaml`), не
подстройка «на глазок».

**Критерий успеха.** Может изменить порог или политику предсказуемо и
увидеть эффект через регрессию (`/judge-regress`); ошибка в конфиге — это
внятное сообщение с локацией, а не traceback и не молчание.

## Кто на какие гейты смотрит

| Гейт                          | Новичок | Эксперт | Владелец системы | Админ |
| ----------------------------- | :-----: | :-----: | :--------------: | :---: |
| 01 — статика                  |    ✓    |    ✓    |                  |   ✓   |
| 02 — права (allowlist)        |    ✓    |    ✓    |        ✓         |   ✓   |
| 03 — дубли                    |         |    ✓    |                  |   ✓   |
| 04 — рубрика                  |         |         |                  |   ✓   |
| 05 — токен-эффективность      |         |         |                  |   ✓   |
| 06 — redteam                  |         |         |    ✓ (send/delete)|   ✓   |
| 07 — цепочки (`uses:`)        |         |    ✓    |                  |   ✓   |
| 08 — ссылки на файлы          |         |    ✓    |                  |   ✓   |

Новичок практически всегда останавливается на 01–02. Эксперт живёт вокруг
02 и 07 (композиция группы). Владелец системы отвечает за политику гейта 02
для своей системы (и за то, какие её инструменты попадают под
`always_flag_for_review` гейта 06). Админ надзирает за всеми гейтами, но
особенно за судье-зависимыми (04/05/06) и их порогами.
