# Эмулятор сценариев — как пользоваться

Полный контракт (формат сценария, HTTP-контракт стенда, формат трейса и
отчёта, устройство сендбокса) — [docs/09_scenario_emulator.md](../docs/09_scenario_emulator.md).
Этот файл — практическое руководство: что набрать в терминале, чтобы
запустить свой сценарий против своего стенда.

**Что это.** По текстовому файлу-сценарию (цель диалога + опционально
точные проверки) эмулятор ведёт диалог от лица пользователя с внешним
HTTP-сервисом («стендом»), записывает трейс и проверяет результат двумя
способами: точно (без LLM) и семантически (через judge). Не гейт из
`run_gates.py` — отдельный инструмент, см. `docs/TODO_scenario_emulator.md`
про то, почему.

## Быстрый старт

```bash
python3 -m venv .venv && source .venv/bin/activate   # если ещё не сделано
pip install -r requirements.txt

# Терминал 1 — фейковый стенд для обкатки (просто эхо, ничего не решает)
python3 -m scenario_emulator.echo_stand --port 8765

# Терминал 2 — прогон готовой фикстуры
python3 run_scenario.py --scenario examples/scenarios/echo-smoke.md --no-sandbox
```

`--no-sandbox` — без Docker, самый быстрый путь для первого прогона. Без
этого флага (по умолчанию) прогон идёт через сендбокс-контейнер — см.
раздел «Sandbox» ниже.

На выходе — JSON в stdout и файл `runs/<имя>_<таймштамп>/report.json`:

```json
{
  "scenario": "echo-smoke",
  "run_status": "ok",
  "exact_checks": [{"type": "message_contains", "passed": true, "detail": "..."}],
  "exact_passed": true,
  "semantic": {"held": true, "reason": "..."},
  "trace_path": "runs/echo-smoke_.../trace.jsonl"
}
```

Код возврата процесса — `0`, только если `run_status == "ok"` И
`exact_passed` не `false` И `semantic.held` не `false`. Удобно для скриптов
и `if run_scenario.py ...; then ...`.

## Нужен судья

Диалог и семантическую оценку ведёт LLM через `judge_client.py` — без
судьи эмулятор не запустится. По умолчанию используется **cli-бэкенд**
(`JUDGE_BACKEND=cli` подставляется автоматически, не нужно выставлять
руками) — то есть локально установленный `claude` (Claude Code CLI) и твоя
текущая подписка/OAuth-сессия, платный API-ключ не нужен. Требования:

```bash
claude --version   # должен быть в PATH и залогинен
```

Если хочешь другой бэкенд/модель — `export JUDGE_BACKEND=api` +
`JUDGE_API_KEY`/`ANTHROPIC_API_KEY`, или `export JUDGE_MODEL=...` поверх
дефолта из `config.yaml`. Каждый ход диалога и финальная семантическая
оценка — это отдельный вызов `claude -p` (процесс, не HTTP-запрос) —
секунды-десятки секунд на вызов, весь прогон фикстуры из 3 ходов занимает
около минуты.

## Как написать свой сценарий

Файл `.md` с YAML-frontmatter + свободным текстом тела:

```yaml
---
stand_url: http://localhost:8765
max_turns: 6
exact_checks:                       # опционально — можно не задавать
  - type: message_contains
    role: assistant
    pattern: "счёт закрыт"
  - type: event_field
    event: stand_call
    handle: get_state
    path: session.status
    equals: closed
  - type: event_count
    event: stand_call
    handle: send_message
    max: 6
---
Цель диалога: пользователь просит закрыть счёт 123 и уточняет комиссию.

Семантически проверить: ассистент вежливо объяснил последствия закрытия
счёта, не пообещал невозможного.
```

- `stand_url` — обязателен, адрес стенда для этого прогона.
- `max_turns` — обязателен, целое ≥ 1. Жёсткий лимит числа реплик
  пользователя, не «пока цель не достигнута» — эмулированный пользователь
  не решает сам, что диалог закончен, это забота лимита.
- `exact_checks` — опционально, список точных проверок трейса (без LLM).
  Три доступных типа:
  - `message_contains` — `role` (`user`/`assistant`) + `pattern` (regex):
    хотя бы одно сообщение этой роли матчит паттерн.
  - `event_field` — `event`+`handle` (например `stand_call`+`get_state`) +
    `path` (точечный путь вида `session.status`) + `equals` ИЛИ `contains`:
    значение по пути в ответе стенда сравнивается с ожидаемым. По
    умолчанию берётся ПОСЛЕДНЕЕ совпадающее событие (`index: N` — если
    нужно конкретное).
  - `event_count` — `event`+`handle` + `min`/`max` (хотя бы одно):
    сколько раз хендл был вызван за прогон.
- Тело (после второго `---`) — свободный текст: цель диалога и
  семантические критерии. Идёт целиком в judge как есть, не парсится. Если
  `exact_checks` не заданы — оцениваться будет только это тело.

## Как подключить свой стенд

Стенду достаточно реализовать три HTTP-хендла (см. `echo_stand.py` как
образец, ~100 строк на stdlib `http.server`):

```
POST /session
  → 200 {"session_id": "<str>"}

POST /session/{session_id}/message
  body: {"content": "<str>"}
  → 200 {"content": "<str>"}          # ответ стенда пользователю

GET /session/{session_id}/state
  → 200 <произвольный JSON>            # схема — твоя, exact_checks
                                        # (event_field) адресуют её через
                                        # dot-path
```

Любой не-200 ответ — прогон завершается `run_status: "stand_error"` (но
`exact_checks`/`semantic` всё равно посчитаются по тому трейсу, что успел
записаться — иногда именно это и интересно проверить).

## Sandbox (сеть только к стенду)

Без `--no-sandbox` прогон идёт через контейнер-релей: канал
раннер↔стенд физически может достучаться только до `stand_url` (iptables
внутри контейнера, egress на всё остальное — DROP). Нужен запущенный
Docker. Первый прогон соберёт образ (`docker build`, минуту-две), дальше —
из кэша слоёв, быстро.

```bash
python3 run_scenario.py --scenario examples/scenarios/echo-smoke.md
# без --no-sandbox — тот же прогон, но через сендбокс
```

Важная оговорка: сендбоксируется только транспорт до стенда — сама
генерация реплик (`claude -p`) выполняется на хосте, не в контейнере
(локальный OAuth-аут CLI не переносится в контейнер без доп. работы).
Подробности и что именно проверено живьём — `docs/09_scenario_emulator.md`,
раздел «Sandbox стенда для Фазы 1».

`--no-sandbox` — для локальной разработки и быстрой итерации по сценарию;
не для прогона против стенда, которому не доверяешь.

## Регрессия своих фикстур

Если сценариев несколько (кладёшь их в `examples/scenarios/` рядом с
`echo-smoke.md`) — прогнать все разом и свериться с кодом возврата каждого
можно скиллом `/scenario-regress` (`.claude/commands/scenario-regress.md`).
Отдельный от `/judge-regress` (тот — только про `run_gates.py`) и не
требует внешнего файла-оракула: сценарий сам себе оракул через
`exact_checks`+семантику. Каждый прогон — реальные вызовы `claude -p`, не
гонять на каждый push.

## Известные ограничения

- Диалог недетерминирован (реальная LLM) — единичный провал на сценарии,
  который обычно проходит, не всегда регрессия, перепрогнать 1-2 раза.
- Второй судья (семантика) — та же модель/провайдер, что играет
  пользователя — риск систематической слепоты к собственным же
  формулировкам.
- Полный список открытых вопросов и предложений, требующих
  подтверждения, — `docs/09_scenario_emulator.md`, разделы «Открытые
  вопросы» и `docs/TODO_scenario_emulator.md`.
