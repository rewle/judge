# Эмулятор сценариев — как пользоваться

Полный контракт (формат сценария, HTTP-контракт стенда, формат трейса и
отчёта, устройство сендбокса) — [docs/09_scenario_emulator.md](../docs/09_scenario_emulator.md).
Этот файл — практическое руководство: что набрать в терминале, чтобы
запустить свой сценарий против своего стенда.

**Что это.** По текстовому файлу-сценарию (цель диалога + опционально
точные проверки) эмулятор ведёт диалог от лица пользователя с внешним
HTTP-сервисом («стендом»), записывает трейс и проверяет результат двумя
способами: точно (без LLM) и семантически (через judge). Не гейт из
`run_gates.py` — отдельный инструмент, см. `docs/TODO_scenario_emulator.md`
про то, почему.

## Быстрый старт

```bash
python3 -m venv .venv && source .venv/bin/activate   # если ещё не сделано
pip install -r requirements.txt

# Терминал 1 — фейковый стенд для обкатки (просто эхо, ничего не решает)
python3 -m scenario_emulator.echo_stand --port 8765

# Терминал 2 — прогон готовой фикстуры
python3 run_scenario.py --scenario examples/scenarios/echo-smoke.md --no-sandbox
```

`--no-sandbox` — без Docker, самый быстрый путь для первого прогона. Без
этого флага (по умолчанию) прогон идёт через сендбокс-контейнер — см.
раздел «Sandbox» ниже.

На выходе — JSON в stdout и файл `runs/<имя>_<таймштамп>/report.json`:

```json
{
  "scenario": "echo-smoke",
  "run_status": "ok",
  "exact_checks": [{"type": "message_contains", "passed": true, "detail": "..."}],
  "exact_passed": true,
  "semantic": {"held": true, "reason": "..."},
  "trace_path": "runs/echo-smoke_.../trace.jsonl"
}
```

Код возврата процесса — `0`, только если `run_status == "ok"` И
`exact_passed` не `false` И `semantic.held` не `false`. Удобно для скриптов
и `if run_scenario.py ...; then ...`.

## Нужен судья

Диалог и семантическую оценку ведёт LLM через `judge_client.py` — без
судьи эмулятор не запустится. По умолчанию используется **cli-бэкенд**
(`JUDGE_BACKEND=cli` подставляется автоматически, не нужно выставлять
руками) — то есть локально установленный `claude` (Claude Code CLI) и твоя
текущая подписка/OAuth-сессия, платный API-ключ не нужен. Требования:

```bash
claude --version   # должен быть в PATH и залогинен
```

Если хочешь другой бэкенд/модель — `export JUDGE_BACKEND=api` +
`JUDGE_API_KEY`/`ANTHROPIC_API_KEY`, или `export JUDGE_MODEL=...` поверх
дефолта из `config.yaml`. Каждый ход диалога и финальная семантическая
оценка — это отдельный вызов `claude -p` (процесс, не HTTP-запрос) —
секунды-десятки секунд на вызов, весь прогон фикстуры из 3 ходов занимает
около минуты.

## Как написать свой сценарий

Файл `.md` с YAML-frontmatter + свободным текстом тела:

```yaml
---
stand_url: http://localhost:8765
max_turns: 6
exact_checks:                       # опционально — можно не задавать
  - type: message_contains
    role: assistant
    pattern: "счёт закрыт"
  - type: event_field
    event: stand_call
    handle: get_state
    path: session.status
    equals: closed
  - type: event_count
    event: stand_call
    handle: send_message
    max: 6
---
Цель диалога: пользователь просит закрыть счёт 123 и уточняет комиссию.

Семантически проверить: ассистент вежливо объяснил последствия закрытия
счёта, не пообещал невозможного.
```

- `stand_url` — обязателен, адрес стенда для этого прогона.
- `max_turns` — обязателен, целое ≥ 1. Жёсткий лимит числа реплик
  пользователя, не «пока цель не достигнута» — эмулированный пользователь
  не решает сам, что диалог закончен, это забота лимита.
- `compose_file` — опционально, путь к docker-compose файлу относительно
  корня репозитория. Если задан, `run_scenario.py` сам поднимает стенд
  (`docker compose up -d --build`) перед прогоном и убирает (`down`)
  после — ничего не нужно поднимать вручную. Стенд обязан отвечать на
  `GET /health` (раннер поллит его до готовности, таймаут 30 с). Пример —
  `open-deposit.md` ниже.
- `exact_checks` — опционально, список точных проверок трейса (без LLM).
  Три доступных типа:
  - `message_contains` — `role` (`user`/`assistant`) + `pattern` (regex):
    хотя бы одно сообщение этой роли матчит паттерн.
  - `event_field` — `event`+`handle` (например `stand_call`+`get_state`) +
    `path` (точечный путь вида `session.status`) + `equals` ИЛИ `contains`:
    значение по пути в ответе стенда сравнивается с ожидаемым. По
    умолчанию берётся ПОСЛЕДНЕЕ совпадающее событие (`index: N` — если
    нужно конкретное).
  - `event_count` — `event`+`handle` + `min`/`max` (хотя бы одно):
    сколько раз хендл был вызван за прогон.
- Тело (после второго `---`) — свободный текст: цель диалога и
  семантические критерии. Идёт целиком в judge как есть, не парсится. Если
  `exact_checks` не заданы — оцениваться будет только это тело.

## Примеры

Пять готовых фикстур в `examples/scenarios/`:

- **`echo-smoke.md`** — смоук-тест механизма (Фаза 1), без бизнес-логики:
  стенд (`echo_stand.py`) просто повторяет реплику пользователя с префиксом
  `echo: `. Показывает минимальный контракт, ничего содержательного.
- **`close-account.md`** — реалистичный пример: диалог о закрытии счёта 123
  (тот же сюжет, что в примере из `docs/09_scenario_emulator.md`,
  "Формат сценария"). Стенд (`close_account_stand.py`, порт 8766) держит
  простое состояние счёта `open → pending_confirmation → closed`, объясняет
  комиссию и не даёт закрыть счёт без явного подтверждения. Сценарий сочетает
  все три типа `exact_checks` (упоминание комиссии, `account.status ==
  closed` через `get_state`, потолок числа сообщений) и семантическую
  проверку («не пообещал невозможного» — что уже проведённые платежи можно
  отменить). Используй его как шаблон при написании своего сценария —
  ближе к реальному кейсу, чем `echo-smoke.md`.

```bash
python3 -m scenario_emulator.close_account_stand --port 8766
python3 run_scenario.py --scenario examples/scenarios/close-account.md --no-sandbox
```

Проверено живым прогоном 2026-07-15 (`--no-sandbox`, реальный `claude -p`):
`exact_passed: true`, `semantic.held: true`, код возврата 0.

- **`open-deposit.md`** — референс паттерна детерминированного
  подтверждения (widget/action вместо текста, схема архитектуры —
  [docs/assets/open-deposit-architecture.html](../docs/assets/open-deposit-architecture.html)):
  по мотивам реального
  банковского ассистента, где клик по кнопке «Подтвердить» превращается
  харнессом в свободный текст, летящий в модель, — риск, что модель
  «глюкнёт» при интерпретации. Стенд здесь — три отдельных сервиса
  (`scenario_emulator/open_deposit/`): `ui_gateway` (порт 8767, transport
  passthrough, единственный опубликованный наружу), `harness`
  (маршрутизация: текст → rule-based заглушка, `action: confirm` → сразу
  тул в обход модели) и `tool` (реестр `confirmation_id`, держатель
  истины — открывает вклад только по реально зарегистрированному и ещё не
  использованному id). Харнесс никогда не открывает вклад по тексту —
  даже дословное «Подтверждаю открытие вклада» получает новый запрос
  подтверждения виджетом, а не открытие. Сценарий проверяет ровно это:
  `deposit.opened_via == widget_confirm` и хотя бы одно событие
  `widget_confirm`/`auto_confirm` в трейсе (раннер сам подставляет
  `action: confirm` на втором ходу, судью не спрашивает). Расширение
  контракта (`widget` в ответе, `action`/`confirmation_id` в запросе,
  `compose_file`) — см. `docs/09_scenario_emulator.md`, аддендум
  "Расширение контракта: widget-подтверждение и compose-стенды".

  ```bash
  python3 run_scenario.py --scenario examples/scenarios/open-deposit.md --no-sandbox
  ```

  Одна команда — `compose_file` во frontmatter заставляет `run_scenario.py`
  самому поднять и убрать три контейнера. В отличие от `echo-stand`/
  `close-account-stand`, этот стенд не поднимается bare-процессом — только
  через compose.

  Проверено живым прогоном 2026-07-16 (`--no-sandbox` и через `--sandbox`,
  реальный `claude -p`): `exact_passed: true`, `semantic.held: true`, код
  возврата 0 в обоих режимах; трейс подтверждает, что шаг подтверждения
  записан как `widget_confirm`, а не `user_message`.

  **Демо в браузере** (для ручного показа, без judge/LLM-эмулятора):
  ```bash
  docker compose -f docker/open-deposit-compose.yml up -d --build
  open http://localhost:8767/   # или просто открыть URL в браузере
  docker compose -f docker/open-deposit-compose.yml down   # когда закончили
  ```
  Простой чат: печатаешь реплики сам (LLM здесь не участвует), при просьбе
  открыть вклад появляется карточка подтверждения с настоящей кнопкой —
  клик шлёт `{"action":"confirm",...}` напрямую в JS, минуя любое текстовое
  поле. Внизу страницы — сворачиваемая панель с сырыми запросами/ответами,
  наглядно показывающая, что клик и текст — разные пути. Дословное
  «Подтверждаю открытие вклада», набранное текстом, вклад не откроет —
  тот самый кейс со скриншота, который стенд не должен пропускать.

  **Как добавить ещё одну операцию в тот же harness/tool.** Стенд умеет
  три операции: открытие (`open_deposit`), закрытие (`close_deposit`) и
  пополнение (`top_up_deposit`) — две последние добавлены 2026-07-17 как
  проверка расширяемости, используй их код как референс. Механизм общий:
  `/register` в `tool.py` принимает обязательное поле `kind`, хендлер
  `POST /open_deposit` (URL исторический, НЕ переименовывался — его зовёт
  `harness._confirm()` по имени; логика внутри общая) читает `kind` из
  зарегистрированной записи, делает соответствующую мутацию и возвращает
  `{"deposit": ..., "operation": <kind>}`. Чтобы добавить операцию N+1,
  не трогая `ui_gateway.py` и существующие фикстуры:

  1. **Ветка `kind` в `tool.py`.** В хендлере `/open_deposit` добавь ветку
     для нового `kind` (валидация предусловий → 409, мутация `_DEPOSITS`,
     ответ с `operation`). Референс: ветки `close_deposit` (депозита нет
     или не `opened` → 409, иначе `status = closed`) и `top_up_deposit`
     (плюс валидация `top_up_amount` из `details`). Без этого шага харнесс
     физически не может сказать тулу, ЧТО подтверждать.

  2. **Ветка-триггер в `harness._reply()` + текст ответа в `_confirm()`.**
     Новая ветка в `_reply()` регистрирует `confirmation_id` со своим
     `kind` и возвращает свой `widget` — референс: ветки «закр» и «пополн»
     (осторожно со словоформами триггера: `"закры"` не матчит «закрой»,
     бери минимальную общую основу). В `_confirm()` маршрутизация не
     меняется (она не знает и не должна знать тип операции заранее), но
     текст ответа пользователю строится по полю `operation` из ответа
     тула — добавь свою ветку текста, иначе после подтверждения новой
     операции пользователь получит сообщение про открытие. Критический
     инвариант сохраняется по построению для любой операции: текстовый
     путь только регистрирует и просит виджет, мутация состояния —
     исключительно через `action:confirm`.

  3. **Демо-страница править не нужно** — `addWidgetCard()` в
     `ui_gateway.py` уже рендерит любой `widget.details` общим кодом,
     новая операция появится в чате автоматически. Оговорка: подписи полей
     карточки («Вклад», «Сумма», «Срок», «Ставка», «Списание») сейчас
     захардкожены под депозит — если операция несёт другие детали или
     подпись должна быть другой, эти строки в `addWidgetCard()` тоже
     придётся тронуть.

  4. **Новая фикстура, не трогая существующие.** Референс —
     `deposit-close-flow.md` (открыть → подтвердить → закрыть →
     подтвердить: `max_turns: 4`, `deposit.status: closed`,
     `event_count widget_confirm min: 2`) и `deposit-topup-flow.md`
     (то же с пополнением, проверяет `deposit.details.amount` после).
     `stand_url`/`compose_file` — те же, стенд общий. Имя файла выбирай
     не пересекающимся по смыслу с `close-account.md` (другой механизм).

  5. **Проверить.** Сначала руками curl'ом (как в `docs/09_scenario_emulator.md`
     — текст-триггер → widget → дословное «Подтверждаю …» текстом → стейт
     НЕ изменился → `action:confirm` → стейт изменился → повторный confirm
     отклонён), потом `run_scenario.py --scenario <новый файл>
     --no-sandbox`. Обязательно перепрогнать и существующие фикстуры
     стенда (`open-deposit.md`, `deposit-close-flow.md`,
     `deposit-topup-flow.md`) — новая ветка не должна менять их
     поведение, но это стоит проверить, а не предполагать.

- **`deposit-close-flow.md`** и **`deposit-topup-flow.md`** — вторая и
  третья операции того же стенда `open-deposit` (закрытие и пополнение),
  добавлены 2026-07-17 как живая проверка инструкции по расширению выше:
  два виджет-подтверждения за прогон (`max_turns: 4`,
  `event_count widget_confirm min: 2`), тот же `compose_file`. Проверены
  живыми прогонами вместе с регрессией `open-deposit.md`.

## Как подключить свой стенд

Стенду достаточно реализовать три HTTP-хендла (см. `echo_stand.py` как
образец, ~100 строк на stdlib `http.server`):

```
POST /session
  → 200 {"session_id": "<str>"}

POST /session/{session_id}/message
  body: {"content": "<str>"}
  → 200 {"content": "<str>"}          # ответ стенда пользователю

GET /session/{session_id}/state
  → 200 <произвольный JSON>            # схема — твоя, exact_checks
                                        # (event_field) адресуют её через
                                        # dot-path
```

Любой не-200 ответ — прогон завершается `run_status: "stand_error"` (но
`exact_checks`/`semantic` всё равно посчитаются по тому трейсу, что успел
записаться — иногда именно это и интересно проверить).

Расширение контракта (widget-подтверждение через `action`/
`confirmation_id`, поле `compose_file`) — см.
`docs/09_scenario_emulator.md`, аддендум "Расширение контракта:
widget-подтверждение и compose-стенды" и пример `open-deposit.md` выше.

## Sandbox (сеть только к стенду)

Без `--no-sandbox` прогон идёт через контейнер-релей: канал
раннер↔стенд физически может достучаться только до `stand_url` (iptables
внутри контейнера, egress на всё остальное — DROP). Нужен запущенный
Docker. Первый прогон соберёт образ (`docker build`, минуту-две), дальше —
из кэша слоёв, быстро.

```bash
python3 run_scenario.py --scenario examples/scenarios/echo-smoke.md
# без --no-sandbox — тот же прогон, но через сендбокс
```

Важная оговорка: сендбоксируется только транспорт до стенда — сама
генерация реплик (`claude -p`) выполняется на хосте, не в контейнере
(локальный OAuth-аут CLI не переносится в контейнер без доп. работы).
Подробности и что именно проверено живьём — `docs/09_scenario_emulator.md`,
раздел «Sandbox стенда для Фазы 1».

`--no-sandbox` — для локальной разработки и быстрой итерации по сценарию;
не для прогона против стенда, которому не доверяешь.

## Регрессия своих фикстур

Если сценариев несколько (кладёшь их в `examples/scenarios/` рядом с
`echo-smoke.md`) — прогнать все разом и свериться с кодом возврата каждого
можно скиллом `/scenario-regress` (`.claude/commands/scenario-regress.md`).
Отдельный от `/judge-regress` (тот — только про `run_gates.py`) и не
требует внешнего файла-оракула: сценарий сам себе оракул через
`exact_checks`+семантику. Каждый прогон — реальные вызовы `claude -p`, не
гонять на каждый push.

## Известные ограничения

- Диалог недетерминирован (реальная LLM) — единичный провал на сценарии,
  который обычно проходит, не всегда регрессия, перепрогнать 1-2 раза.
- Второй судья (семантика) — та же модель/провайдер, что играет
  пользователя — риск систематической слепоты к собственным же
  формулировкам.
- Полный список открытых вопросов и предложений, требующих
  подтверждения, — `docs/09_scenario_emulator.md`, разделы «Открытые
  вопросы» и `docs/TODO_scenario_emulator.md`.
