# Регрессия сценарных фикстур

Цель: заменить ручной прогон `run_scenario.py` по каждому файлу в
`examples/scenarios/` на автоматический — без этого шага легко пропустить,
что правка в `scenario_emulator/*.py` (раннер, exact-checker, семантика)
сломала фикстуру, которая раньше проходила.

В отличие от `judge-regress` (`.claude/commands/judge-regress.md`), здесь
не нужен отдельный оракул: каждая сценарная фикстура уже несёт своё
ожидание внутри себя — `exact_checks` во frontmatter и критерии в теле
(оценивает семантический судья). "Правильный" результат фикстуры —
`run_scenario.py` вернул код 0. Фикстуры, которые ПРОВЕРЯЮТ провальный
путь (например будущая фикстура с намеренно недостижимым `exact_checks`),
должны сами это обозначить в комментарии/имени файла — см. "Типичные
ловушки" ниже.

## Шаг 1. Поднять фейковый эхо-стенд, прогнать все фикстуры, собрать отчёты

```bash
cd /Users/rewle/Projects/judge && source .venv/bin/activate 2>/dev/null

# Стенд для фикстур, объявляющих stand_url http://localhost:8765 — если
# фикстура использует другой стенд, ей нужен отдельный процесс, этот шаг
# его не поднимет.
python3 -m scenario_emulator.echo_stand --port 8765 > /tmp/scenario_regress_stand.log 2>&1 &
STAND_PID=$!
sleep 1

RUN_DIR=$(mktemp -d)
FAILED=0
for scenario in examples/scenarios/*.md; do
  name=$(basename "$scenario" .md)
  echo "=== $name ==="
  python3 run_scenario.py --scenario "$scenario" --no-sandbox --out-dir "$RUN_DIR" \
    > "$RUN_DIR/${name}.stdout" 2> "$RUN_DIR/${name}.stderr"
  code=$?
  echo "exit=$code"
  if [ "$code" != "0" ]; then
    FAILED=1
    echo "--- report.json (или stderr, если прогон не запустился) ---"
    cat "$RUN_DIR/${name}.stdout"
    cat "$RUN_DIR/${name}.stderr" >&2
  fi
done

kill "$STAND_PID" 2>/dev/null
echo
echo "RUN_DIR=$RUN_DIR (report.json/trace.jsonl каждой фикстуры — там, не удалять до разбора)"
[ "$FAILED" = "0" ] && echo "ВСЕ ФИКСТУРЫ ПРОШЛИ" || echo "ЕСТЬ ПРОВАЛЫ — см. вывод выше"
```

## Шаг 2. Сформировать отчёт

- **Если все фикстуры прошли** — одна строка: "N/N фикстур прошли,
  расхождений нет".
- **Если есть провалы** — на каждую упавшую фикстуру:
  - код возврата (`1` — обычный провал exact_checks/semantic; ненулевой,
    но с сообщением "прогон не запустился" в stderr — инфраструктурная
    проблема: нет `claude` в PATH, стенд не поднялся, сендбокс недоступен и
    т.п., не баг раннера/сценария);
  - если это провал проверки — вытащить из `report.json`
    (`$RUN_DIR/<name>.stdout`) конкретные `exact_checks[].detail` с
    `passed: false` и/или `semantic.reason` при `held: false` — то же самое,
    что уже напечатано в выводе Шага 1, просто явно процитировать в отчёте,
    не заставлять читать сырой JSON;
  - трейс конкретного провального прогона — `$RUN_DIR/<name>_*/trace.jsonl`
    (путь есть в `report.json` поле `trace_path`), если для разбора нужен
    полный ход диалога, а не только вердикт.

## Типичные ловушки

- Прогон **недетерминирован** (реальная LLM играет пользователя и судью) —
  единичный провал на фикстуре, которая обычно проходит, не обязательно
  регрессия в коде; перепрогнать 1-2 раза, прежде чем поднимать как баг.
  Стабильный провал на нескольких прогонах подряд — уже настоящий сигнал.
- Скрипт по умолчанию `--no-sandbox` — не проверяет docker/iptables-путь
  (`--sandbox`, дефолт самого `run_scenario.py`). Если правка касалась
  именно `scenario_emulator/docker_sandbox.py`/`sandbox_relay.py`/
  `docker/relay*`, прогнать соответствующую фикстуру ОТДЕЛЬНО без
  `--no-sandbox` — этот скилл сам сендбокс не перепроверяет.
- Если `examples/scenarios/` пуст или фикстура не задаёт `stand_url:
  http://localhost:8765` — Шаг 1 либо ничего не прогонит, либо фикстура
  упадёт с `stand_error` (не значит, что раннер сломан — значит, фикстуре
  нужен другой стенд, который этот скилл не поднимает).
- Каждый прогон — реальные деньги/время (несколько вызовов `claude -p` на
  фикстуру). Не включать в pre-commit/CI на каждый push — только по явному
  запросу или перед коммитом в `scenario_emulator/*.py`.
- Не чинить фикстуры/раннер по результатам этого скилла автоматически —
  только показать; правки отдельным запросом.
