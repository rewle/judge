# judge

CI/CD-гейты для реестра скиллов (Claude Code skills) и конфигов. Каждый PR со
скиллом проходит цепочку гейтов **последовательно**: фейл гейта останавливает
проверку — следующие гейты не запускаются.

## Гейты (в порядке прогона)

| # | Гейт | Детерминизм | Что проверяет | Статус |
|---|---|---|---|---|
| 01 | [static](docs/01_static.md) | детерминированный | frontmatter, длина, пути | реализован |
| 02 | [permissions](docs/02_permissions.md) | детерминированный | tools/MCP против allowlist | реализован |
| 03 | [duplication](docs/03_duplication.md) | v0 детерминированный, v1 (semantic) нет | текстуальные + функциональные дубли | реализован; v1 опционален, требует judge |
| 04 | [rubric](docs/04_rubric.md) | недетерминированный (LLM-judge) | 6 критериев качества + groundedness | реализован, требует judge |
| 05 | [token_efficiency](docs/05_token_efficiency.md) | недетерминированный (behavioral) | токены/латентность vs прирост coverage | реализован, требует judge (только api) |
| 06 | [redteam](docs/06_redteam.md) | статика детерминированная, adversarial-часть нет | устойчивость к prompt injection / расширению прав | реализован, статика без judge, adversarial требует judge |
| 07 | [chain](docs/roadmap_chains.md) | 4 проверки детерминированные, I/O-совместимость нет | граф `uses:`: циклы, битые ссылки, эскалация прав через композицию, необъявленные вызовы в теле, I/O-совместимость (`provides:`) | реализован полностью; реестровый |

Гейты 01-02 дешёвые и детерминированные — гонять на каждый push. Гейт 03
v0-часть тоже дешёвая — гонять всегда; v1-часть и гейты 04-06 дороже (нужен
judge) — см. `docs/` по каждому гейту про рекомендованную частоту. Гейт 07
детерминированная часть дешёвая, I/O-совместимость требует judge; в любом
случае реестровый — не входит в последовательную per-skill цепочку 01-06;
при полном прогоне реестра (`python3 run_gates.py` без `--skill`/`--path`)
запускается автоматически после того, как все скиллы прошли 01-06
(`--skip-chains` отключает).

## Структура

```
gates/        — код гейтов (по одному модулю на гейт)
docs/         — контракт каждого гейта: вход, логика, порог, выход
policies/     — allowlist прав, критерии рубрики, библиотека атак
examples/     — скиллы-фикстуры, каждый рассчитан на провал конкретного гейта
history/      — append-only лог прогонов рубрики (дрифт судьи, см. docs/04)
config.yaml   — порядок гейтов, пороги, пути
run_gates.py  — последовательный раннер (fail-fast)
new_skill.py  — генератор скелета скилла для тех, кто пишет SKILL.md впервые
```

## Пишу свой первый скилл

```bash
python3 new_skill.py my-skill-name          # создаёт ./my-skill-name/SKILL.md
# ...замени все {{...}} на реальный текст...
python3 run_gates.py --path ./my-skill-name  # проверить вне skills_dir
```
Скелет проходит гейт 01 по конструкции, кроме намеренных `{{...}}`-плейсхолдеров
в местах, которые нужно заполнить, — гейт 01 явно перечислит их с номерами
строк. См. docs/01_static.md, раздел "Защита от новичков".

**Гейты не запускаются сами по `git commit`.** В этом репозитории нет
git-хука и нет CI-воркфлоу — `judge` это библиотека гейтов, которую целевой
реестр подключает к своему пайплайну. Пока это не сделано, единственная
проверка — запустить `run_gates.py` вручную (см. выше). Чтобы гейты ловили
проблемы до ревью, подключи их в целевой системе одним из способов:

```bash
# pre-commit хук (локально, быстрые детерминированные гейты 01/02/08):
echo 'python3 run_gates.py --path "$(git rev-parse --show-toplevel)/<путь-к-скиллу>" || exit 1' \
  >> .git/hooks/pre-commit && chmod +x .git/hooks/pre-commit

# CI (на PR, вся цепочка, включая judge-гейты с ключом в secrets):
#   run: python3 run_gates.py            # весь реестр + гейт 07
```

## Установка

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
```

## Настройка судьи (гейты 03(v1)/04-06)

Два бэкенда, выбираются через `JUDGE_BACKEND` (по умолчанию `api`).

### api (по умолчанию, нужен для CI)

Настоящий Anthropic API, платный ключ — креды в репозитории не хранятся,
только env:

```bash
export JUDGE_API_KEY=sk-ant-...        # или ANTHROPIC_API_KEY
export JUDGE_BASE_URL=...              # опционально: свой прокси вместо api.anthropic.com
export JUDGE_MODEL=claude-sonnet-5     # опционально, дефолт см. config.yaml -> judge.model
```

Без ключа гейты 04-05 возвращают `not_configured` (не блокируют цепочку, но и
не пропускают молча — статус явно виден в отчёте). Гейт 05 дополнительно
требует `eval.yaml` рядом со SKILL.md (задача + ожидаемое поведение) — без
него возвращает `skipped`, не `not_configured`. Гейт 06 без ключа всё равно
гоняет статический пред-фильтр (не требует ключа); `not_configured` у него
означает только то, что статика чиста, а adversarial-часть недоступна. Гейт
03 без ключа тоже работает как раньше (только v0, лексика) — v1
(семантическая проверка) молча пропускается, не блокирует и не требует
отдельной настройки сверх той же переменной.

### cli (только для локальной обкатки, без платного ключа)

```bash
export JUDGE_BACKEND=cli
export JUDGE_MODEL=sonnet              # алиас или полное имя модели, см. `claude --help`
# опционально: export JUDGE_CLI_BINARY=claude  (если бинарник не в PATH под этим именем)
```

Под капотом дёргает установленный Claude Code CLI (`claude -p ...
--json-schema ... --no-session-persistence --tools ""`), то есть судья — это
модель текущей подписки/сессии, а не отдельный оплачиваемый API-ключ. Годится
для локальной проверки логики гейтов 03(v1), 04 и 06 (протестировано на
фикстурах: семантический дубль на `spreadsheet-duplicate-finder` /
`dataset-redundancy-detector`, рубрика на `golden-skill`, adversarial-прогон
на скилле с `WebFetch`).

Ограничения:
- **Гейт 05 недоступен** — `claude -p` не даёт offline-эндпоинт подсчёта
  токенов без генерации, а это ядро логики гейта (дешёвый overhead без
  траты токенов). Возвращает `not_configured` с явным объяснением.
- Каждый вызов — отдельный процесс `claude -p` (секунды-десятки секунд), а
  не лёгкий HTTP-запрос — параллельные прогоны (`n_runs=8` в гейте 04)
  ощутимо медленнее и тяжелее, чем через api-бэкенд.
- Не предназначен для CI: использует локальную авторизацию сессии, не
  подходит для воспроизводимого прогона на чужой машине/раннере.

## Запуск

```bash
python3 run_gates.py                       # 01-06 по examples/skills + гейт 07 автоматически, если все прошли
python3 run_gates.py --skill golden-skill  # один скилл, без гейта 07
python3 run_gates.py --skip-chains          # 01-06, без автозапуска гейта 07
python3 run_gates.py --check-chains --registry examples/chains  # только гейт 07, отдельно
```

## Ожидаемые результаты на фикстурах

См. `examples/expected_results.yaml` — какой гейт должен упасть на каком
примере (гейты 01-06). Гейт 07 — отдельный набор фикстур в
`examples/chains/`: `chain-orchestrator`/`chain-worker` (валидный `uses:`
+ совместимый `provides:`, PASS), `chain-cycle-a`/`chain-cycle-b` (цикл,
FAIL), `chain-dangling` (битая ссылка, FAIL),
`chain-escalation-narrow`/`chain-escalation-wide` (эскалация прав через
композицию, FAIL на narrow), `chain-undeclared-caller` (необъявленный
вызов `chain-worker` в теле, FAIL), `chain-io-mismatch-consumer`/
`chain-io-mismatch-provider` (I/O-несовместимость по `provides:`, FAIL на
consumer, требует judge).

## Роадмап

Все гейты 01-06 проверяют один скилл в изоляции. **Гейт 07 (граф
зависимостей skill→skill)** реализован полностью: см.
[docs/roadmap_chains.md](docs/roadmap_chains.md) — конвенция
`uses: [skill-a, skill-b]` во frontmatter (спроектирована для целевых
систем, где конвенцию для skill→skill вызовов задаём мы), гейт проверяет
ацикличность графа, существование ссылок, эскалацию прав через композицию
(объединение `tools:` по цепочке против той же политики, что гейт 02),
необъявленные вызовы в теле и (через отдельное поле `provides:`,
judge-based) I/O-совместимость на стыке скиллов. Реестровая проверка — не
вписывается в `check(skill_path)`, отдельная сигнатура `check_registry()`
и режим `run_gates.py`, встроенный в CI-порядок автоматически. Открыто:
совокупный токен-бюджет цепочки, многошаговый prompt injection и
направление B (skill→file) — не начаты.
